# 코스 가이드라인 파이프라인

이 문서는 한 학습 트랙의 **채점 기준이 되는 정답 자료를 만들고 검증하는 과정**을 정의한다. 정확도 향상 기법과 검증 로직을 자유롭게 더하거나 바꿀 수 있도록, 각 단계를 독립적이고 교체 가능한 형태로 구성한다.

(전체 시스템 구조는 [DESIGN.md](DESIGN.md), 제품 정의는 [SPEC.md](SPEC.md) 참고.)

---

## 1. 목적

화면 결과물이 있는 과제를 자동으로 채점하려면 단계마다 "정답 화면"이 있어야 한다. 이 파이프라인은 학습 계약서(syllabus)가 정해진 뒤 **단계별 정답 구현과 정답 화면을 만들고 검증해 숨겨 보관**한다. 학습자는 정답 코드를 절대 보지 못하며, 자신의 코드는 100% 직접 작성한다.

이 과정은 트랙을 준비할 때 한 번 실행되고 자주 돌지 않는다. 정확성이 가장 중요하므로 비용이 큰 기법도 적극 사용한다.

---

## 2. 입력과 출력

- **입력** — 학습 계약서(단계별 범위·합격 기준), 참고 자료(원본 코드·논문·강의 자료), 단계별 고정 환경(모델·텍스처·카메라·해상도). 이 환경 정의는 학습자 스타터 템플릿·서버 캡처와 동일한 것을 공유한다(어긋나면 채점 비교가 무의미하다).
- **출력** — 단계별 정답 화면(채점 비교 기준) + 그 화면을 만든 정답 구현(숨김 보관, 학습자 비공개).

---

## 3. 전체 흐름

```
학습 계약서 확정
   │
   ▼
단계마다:  정답 생성 ──▶ 정답 검증 ──▶ 통과하면 정답 화면을 숨겨 보관
                                  └ 실패하면 다시 생성
```

생성과 검증은 각각 **여러 독립 기법의 묶음**이다. 새 기법을 더하거나 빼도 나머지에 영향이 없도록 구성한다.

---

## 4. 정답 생성

올바른 정답 구현을 만들기 위해 다음을 조합한다.

- **참고 자료 기반 작성** — 기억이 아니라 트랙의 참고 자료(원본 알고리즘·논문)를 근거로 작성한다.
- **충분한 연산 투입** — 한 번만 도는 작업이므로 가장 강한 모델과 충분한 추론을 사용한다.
- **빌드·렌더 자가 수정** — 작성 → 빌드 → 실행·렌더 → 오류 확인 → 수정을 반복한다. 컴파일·실행 오류는 여기서 걸러진다.
- **단계별 점진 작성** — 전체를 한 번에 만들지 않고 마일스톤 단위로 작게 만든다.
- **기존 구현 우선** — 신뢰할 수 있는 기존 구현이 있으면 새로 만들기보다 범위에 맞춰 활용한다.

---

## 5. 정답 검증

생성된 정답 화면이 실제로 옳은지 여러 층으로 확인한다. 각 검증은 독립적이며, 통과 기준과 항목은 조정할 수 있다.

- **반복 생성 일치** — 정답을 여러 번 독립적으로 만들어 결과 화면들이 서로 일치하는지 본다. 갈리면 신뢰할 수 없으므로 사람 확인으로 넘긴다.
- **원본 대조** — 알려진 원본(유명 게임 등)이 있으면 해당 범위에 한해 비교한다.
- **수학적 불변식** — 결과가 만족해야 할 성질(예: 대칭, 거리-크기 비례)을 자동 검사한다.
- **기본 점검** — 빈 화면이 아님, 기대한 형태가 있음, 색 분포가 정상 범위임 등을 자동 확인한다.
- **교차 검토** — 별도의 검토 과정으로 정답 구현의 논리 오류를 점검한다.
- **최종 사람 확인** — 위를 통과한 정답 화면을 사람이 한 번 눈으로 확인한다. 화면만 보므로 학습에 영향을 주지 않는다.

---

## 6. 유연성 원칙

정확도는 계속 개선되어야 하므로 다음을 지킨다.

- 생성·검증의 각 기법은 **독립 단계**로 두어, 더하거나 빼거나 순서를 바꿔도 나머지가 영향받지 않게 한다.
- 검증의 통과 기준값(유사도 임계값 등)은 **설정으로 분리**해, 코드를 고치지 않고도 조정한다.
- 한 번 통과해 고정된 정답은 그대로 보관하며, 과제 범위가 바뀔 때만 다시 만든다.
