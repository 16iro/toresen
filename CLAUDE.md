# Toresen — 프로젝트 컨텍스트

이 파일은 어느 기기에서든 이 저장소를 열었을 때 작업 맥락과 규칙을 바로 이어받기 위한 것이다.

## 한 줄 요약

AI가 "깐깐한 시니어 트레이너"로 빙의해 GitHub PR 기반으로 코드 리뷰와 과제를 주는 **자기주도 코딩 훈련 도장**. AI는 코드를 대신 짜지 않고 리뷰·과제 발급만 한다. 첫 콘텐츠 주제는 그래픽스, 로컬-퍼스트(자체 호스팅) 구조이며, LLM 추론 비용은 사용자 본인 구독으로 부담한다.

## 문서 지도

- **SPEC.md** — 제품 정의 (무엇을 / 왜)
- **DESIGN.md** — 기술 설계 (어떻게: 전체 구조 · 실행 아키텍처 · 데이터/상태 · 리뷰 · GitHub · 온보딩)
- **COURSE_GUIDELINE_PIPELINE.md** — 정답 자료(골든) 생성·검증 (유연하게 교체 가능)
- **ROADMAP.md** — MVP 범위와 빌드 마일스톤

## 작업 규칙 (이 프로젝트에서 함께 일할 때)

- **대화는** 문서체 말고 직장동료에게 말하듯 편한 구어체(친근한 해요체)로. 표·과한 헤더·불릿 나열을 기본 포맷으로 쓰지 않는다.
- **대화는 짧게.** 핵심만 간결하게 요약하고, 더 물으면 그때 단계적으로 깊이 들어간다.
- **산출 문서(SPEC/DESIGN 등)는** 처음 보는 사람이 이해할 수 있게 쓴다. 우리끼리 쓰던 약어·결정 코드, 진행 마커, "지난 버전 대비" 같은 메타 표현, 선택지·근거 같은 의사결정 파편을 남기지 않는다. 최종본만 선언적으로, 큰 구조에서 세부로 분할정복하며 기술한다.

## 현재 상태 (2026-06-23 기준)

- 설계 결정은 사실상 완료. 여러 차례 교차검증으로 **"MVP 견적 가능(착수 가능)"** 판정.
- 남은 미해결은 결정이 아니라 **검증 스파이크**뿐: 화면 없는 컨테이너에서 GPU 렌더가 되는지, 파이프라인 엔진을 직접 만들지 기존 도구를 쓸지.
- 웹 대시보드 디자인 초안 검토 완료. 제출 플로우·실제 온보딩(본인 AI 도구 연결)·강제 통과 톤다운·자동 캡처·상태 화면이 반영됨. 남은 디자인 자잘: 오타("캡처"), 보상용 레벨/XP와 내부 난이도 보정 레벨의 용어 분리.
