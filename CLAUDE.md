# Toresen — 프로젝트 컨텍스트

이 파일은 어느 기기에서든 이 저장소를 열었을 때 작업 맥락과 규칙을 바로 이어받기 위한 것이다.

## 한 줄 요약

AI가 "깐깐한 시니어 트레이너"로 빙의해 GitHub PR 기반으로 코드 리뷰와 과제를 주는 **자기주도 코딩 훈련 도장**. AI는 코드를 대신 짜지 않고 리뷰·과제 발급만 한다. **공용 호스팅 서비스**로 운영되며, 개인이 자신의 수요에 맞춰 **강의실(트랙)을 직접 생성**하고 공개 강의실을 **포크**해 가져온다(트랙=레포 모델). 첫 시드 예시 주제는 그래픽스. 무거운 연산(추론·빌드·렌더)의 실행 위치는 **연산 모드**로 정한다 — **모드 A**(기본): 사용자 본인 환경에서 실행하고 비용도 본인 부담(BYO), **모드 B**(선택): 운영자가 서버에서 수행해 공개 신뢰 신호를 보증. **구현은 모드 A를 기준으로 한다.**

## 문서 지도

기획·설계 문서는 [docs/](docs/)에 모아 둔다.

- **docs/SPEC.md** — 제품 정의 (무엇을 / 왜)
- **docs/DESIGN.md** — 기술 설계 (어떻게: 전체 구조 · 실행 아키텍처 · 데이터/상태 · 리뷰 · GitHub · 온보딩 · 상위 화면)
- **docs/COURSE_GUIDELINE_PIPELINE.md** — 골든(정답 판정 자료) 생성·검증의 **공통 베이스** + 도메인 모듈 인터페이스·색인
- **docs/golden_pipelines/** — 도메인별 골든 모듈. `graphics.md`(MVP 활성). 알고리즘·웹은 계획
- **docs/CODE_REVIEW_PIPELINE.md** — 코드 리뷰 입력 구성: 레퍼런스 코드베이스 자산화(RDB·VectorDB·KG, 1회·비공개·포크 복제) + 리뷰가 그걸 조회하는 법
- **docs/ROADMAP.md** — MVP 범위와 빌드 마일스톤

## 작업 규칙 (이 프로젝트에서 함께 일할 때)

- **대화는** 문서체 말고 직장동료에게 말하듯 편한 구어체(친근한 해요체)로. 표·과한 헤더·불릿 나열을 기본 포맷으로 쓰지 않는다.
- **대화는 짧게.** 핵심만 간결하게 요약하고, 더 물으면 그때 단계적으로 깊이 들어간다.
- **산출 문서(SPEC/DESIGN 등)는** 처음 보는 사람이 이해할 수 있게 쓴다. 우리끼리 쓰던 약어·결정 코드, 진행 마커, "지난 버전 대비" 같은 메타 표현, 선택지·근거 같은 의사결정 파편을 남기지 않는다. 최종본만 선언적으로, 큰 구조에서 세부로 분할정복하며 기술한다.

## 현재 상태 (디자인 검토 2차 반영 기준)

- 핵심 구조 확정: 상위 레이어(홈=커뮤니티 피드 · 둘러보기 · 내 강의실 · 프로필)와 트랙 앱(대시보드 · 트랙 · 과제 L1/L2/L3 · 트레이너), 그리고 **트랙=레포 포크 모델**(개인 생성 · 공개/포크 · 진도 0 복제 · 출처 표기 · 독립 스냅샷).
- 진단은 **0~10 자가 체크 + 모의 과제 검증**으로 확정. 콘텐츠 난이도(보면상수)·학습자 레벨 추정치는 **사용자에게 철저히 은닉**.
- 동기 설계에서 **연속(streak) 제거**, **학습 열정 온도**(36.5 기준, 리셋·끊김 개념 없음) 도입. 잔디는 사실 기록으로 유지(연속 카운터 없음).
- 온보딩은 **맥락별 분산**: 가입 셋업 마법사(환영·GitHub·트레이너), AI 연결=설정+호출 지점 경고, 레포 등록=강의실 생성 마무리.
- **연산 모드 A/B 전환 구조로 확정.** 빌드·렌더·SSIM·추론의 실행 위치 미결이던 것을 "배포 단위로 모드 A(클라이언트/BYO, 기본) ↔ 모드 B(서버 중앙처리)를 전환, 같은 작업 큐 계약 공유"로 해소. 모드 B는 공개 신뢰 신호(통과자 수 등)를 보증하려는 운영자용이며 ROADMAP M5 옵션. 자가 보고 위조 문제는 커뮤니티 계층(M5)에만 해당하고 핵심 학습 루프는 명예 제도로 충분.
- **렌더 검증 취약성 정리.** "환경 고정=정답 하나" 과잉 주장 → "범위를 좁힘"으로 완화. SSIM 단일 임계 이분법 → 3구간(자동통과/명백한 불일치 자동반려/중간은 멀티모달·본인확인) + 허용 범위(envelope) + 구조 특징 비교. 렌더는 증거지 단독 심판 아님(SPEC 6.3 / DESIGN 4.4·4.5).
- **LLM 리뷰 신뢰성 정리.** 풍부한 컨텍스트(레퍼런스 자산)로 환각률↓ + evidence 실재 기계검증(게시 전 거름, 게이팅 아님) + confidence 표시용 격하 + 이의제기(스코프 재검토: 유지/강등/철회) + override는 드문 최후 비상구 + 코드 누출 필터. **레퍼런스 자산화 파이프라인은 별도 문서**(docs/CODE_REVIEW_PIPELINE.md), 강의실 생성 시 1회 구축·비공개·포크 복제. 리뷰(매 PR)는 그걸 조회만.
- **적응형 난이도 정리.** "잴 수 있는 난이도 숫자"를 버리고 **스킬 트리 경로 위 위치 + 도움(스캐폴딩) 양**으로 다룸. 보면상수=작가가 매긴 노드 순서 티어("이 강의실 기준" 상대 순서, 객관값 아님). 레벨 추정치는 도움 양 컨트롤러일 뿐(이동평균→밴드→낮음/중간/높음). 과제 생성 입력=(다음 노드+도움 양), LLM에 정체불명 난이도 숫자 안 줌. 전부 비노출(SPEC 5.6·8 / DESIGN 3.5).
- **골든 파이프라인 모듈화.** "정답 화면" 전용이던 COURSE를 **베이스(도메인 무관 골든 정의·흐름·검증 층·공개/숨김·모듈 인터페이스)** + **도메인 모듈**(golden_pipelines/graphics.md = MVP, 알고리즘·웹은 계획)로 분리. 골든=정답 판정 자료(이미지/테스트/입출력), 형태·비교법은 강의실 파이프라인 설정. 비시각 도메인의 "제공된 테스트"=골든이 만든 숨긴 채점 자료(DESIGN 4.4).
- 문서(SPEC/DESIGN/CLAUDE/ROADMAP/COURSE+graphics/CODE_REVIEW_PIPELINE) 갱신 완료. 다음 후보 미결: 작업 큐 계약 상세, 데이터 스키마 확정.
