# 강의실 포크 절차와 고려사항

이 문서는 공개 강의실을 **포크할 때 실제로 무엇을 처리하는지(절차)**와 **포크에서 생기는 충돌을 어떻게 다루는지**를 정의한다. 포크의 제품 규칙(공개/포크 토글·진도 0·독립 스냅샷·출처 표기)은 [SPEC.md](SPEC.md) 4장, 복제되는 데이터 항목은 [DESIGN.md](DESIGN.md) 3.1을 따른다.

---

## 1. 무엇이 복제되고 무엇이 리셋되나

**복제한다 (강의실 정의, 스냅샷).**
- 실라버스·스킬 트리, **노드별 고정 과제 명세(요구사항)**, 파이프라인 설정. (과제 명세는 생성 시 확정·골든과 짝이라 그대로 복제되고, 발급 시 스캐폴딩 제시만 학습자별로 달라진다 — DESIGN 3.1·3.5.)
- 골든(정답 구현·검증된 기준 화면) + 그 골든의 **환경 정의**(모델·텍스처·카메라·해상도·렌더 설정).
- 레퍼런스 자산(RDB·VectorDB·KG)을 **스냅샷으로** 복제한다(재빌드하지 않으므로 포크한 사람이 추가 LLM 비용을 지지 않는다).
- 레퍼런스 **출처(인용·포인터)**와 유사도 기준("이 강의실 기준").

**복제하지 않는다.**
- 제3자 레퍼런스 **원본 소스 코드 자체**(예: 둠 소스). 라이선스 때문에 verbatim 재배포하지 않고, 파생물(자산·골든)과 출처 인용만 복제한다. 원작자는 자신이 공개·복제 허용하는 파생 자료에 대한 권리를 보증한다.
- 진척·리뷰·통과 기록·레벨 추정치·학습 열정 온도·활동 기록.

**리셋한다.** 진도는 0에서 시작한다.

## 2. 포크 실행 절차

포크는 "정의 복제"로 끝나지 않는다. 복제만 하면 학습용 레포가 없어 **작동하지 않으므로**, 생성의 마무리 단계(SPEC 5.3)를 포크한 사람에 대해 다시 실행한다.

1. **정의 복제** — 1장의 항목을 스냅샷으로 가져온다.
2. **학습 레포 프로비저닝** — 포크한 사람의 GitHub 계정에 새 저장소를 만들거나(또는 빈 저장소를 지정) **리뷰어 GitHub App을 설치**하고(DESIGN 5.2), 강의실 **스타터 템플릿을 푸시**하며, 복제한 **`syllabus.md`를 그 저장소에 기록**한다.
3. **진도 0 초기화 + 계보 기록** — 진척을 0으로 두고, 부모(출처)와 계보 루트를 기록한다(4장).

포크 전에 **"이 강의실은 ○○가 작성한 빌드 단계를 실행합니다"**를 알리고 작성자·신뢰 신호를 보여준다(낯선 코드를 들이는 것이므로 패키지 설치에 준하는 동의). 복제되는 산출물(스타터 템플릿·골든·자산)은 작성자가 공개한 스냅샷을 **해시로 고정**해 전송 중 변조 없이 그대로 받는다.

완료되면 포크한 사람은 원본과 독립된, **바로 작동하는** 자기 강의실을 갖는다.

## 3. 고려사항과 충돌 처리

- **레포·봇·syllabus 부재** → 2장 절차(레포 프로비저닝)로 해결한다. 포크 시 이 단계를 건너뛰면 강의실은 정의만 있고 돌릴 수 없다.
- **제3자 레퍼런스 라이선스** → 원본 소스를 복제하지 않고 파생물·출처 인용만 복제한다(1장). 라이선스 책임은 원작자에게 있으며, 서비스는 소스 blob을 포크로 재배포하지 않는다.
- **골든의 환경 결합** → 골든은 **정답 구현·환경 정의와 함께** 복제된다. 모드 B는 채점 서버가 정답 구현과 제출물을 **같은 환경에서 렌더**해 비교하므로, 포크한 사람의 GPU·드라이버 차이가 비교에 끼어들지 않는다(작성자 환경 스냅샷 이미지에 의존하지 않는다, DESIGN 4.5). 모드 A 포크는 제출 캡처가 포크한 사람 기기에서 만들어져 서버 기준 화면과 렌더 환경이 갈리며, 이 차이는 허용 범위(envelope)·구조 특징 비교가 흡수한다(개인·안내용 전제).
- **레퍼런스 자산 복제 vs 재빌드** → 스냅샷 복제로 고정한다(재빌드 없음). 골든과 동일한 스냅샷 모델이다.
- **모드가 섞인 포크 계보** → "통과 N명"은 계보 전체에서 **검증된 통과(모드 B)만** 합산한다(SPEC 4장·DESIGN 4.4). 모드 A의 자가 보고 통과는 합산하지 않는다. 따라서 계보에 모드 A·B 강의실이 섞여도 공개 신호는 위조되지 않는다.
- **파이프라인 능력 불일치** → 복제된 파이프라인이 요구하는 실행자를 포크가 놓인 배포가 갖추지 못하면(예: 서버 렌더·비교가 없는 배포의 시각 과제) 해당 단계는 비활성/축소된다. 포크 시 배포 능력과 파이프라인 요구를 대조한다.
- **원본의 이후 변경(골든 수정 포함)** → 포크는 독립 스냅샷이라 받지 않는다. 원본의 잠재 버그도 함께 고정되므로, 원본 갱신을 따라가려면 향후 upstream 동기화(SPEC 11장)를 기다린다.
- **공급망 안전 — 강의실은 "낯선 사람의 코드 패키지"다.** 스타터 템플릿·빌드 설정·레퍼런스 자산은 작성자가 짠 것이고, 포크하면 포크한 사람의 기기(모드 A)나 서버(모드 B)에서 실행·로드된다. 검사로 안전을 증명할 수 없으므로 격리로 막는다:
  - **포크한(남이 만든) 강의실의 빌드·렌더는 격리 샌드박스에서** 수행한다(호스트 파일·자격증명·네트워크 차단). 내가 작성한 강의실의 로컬 빌드는 본인 책임이라 예외다 — 위협은 *낯선 코드*뿐이므로 그 경우만 격리한다(DESIGN 5.3·8장).
  - **작성자의 실행 코드 표면을 줄인다** — 스타터 템플릿은 임의 빌드 훅(Makefile·build.rs·postinstall) 대신 **선언적 빌드 매니페스트**(언어·소스 폴더·플래그)를 두고, 플랫폼의 신뢰된 빌드 러너가 해석한다(DESIGN 5.3).
  - **레퍼런스 자산은 데이터로만 로드한다** — 임의 역직렬화(pickle 등) 금지, 스키마 검증된 데이터로만(실행 가능 형식 배제).
  - 이 격리는 **공개 포크를 켜기 전 필수**다(포크는 ROADMAP M5).

## 4. 계보 모델

- 각 포크는 **부모 포인터**와 **계보 루트 id**를 가진다. 출처 표기는 포크의 포크에서도 계보 내내 유지된다(SPEC 4장).
- "포크 수·통과자 수" 집계는 **계보 루트 기준 쿼리**다. 통과자 수는 검증된 통과(모드 B)만 센다.
- 포크는 독립 레코드이므로, 이 집계는 계보를 가로지르는 쿼리로 산출한다(데이터 스키마에서 계보 루트·부모를 식별자로 둔다).
