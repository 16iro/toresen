# Toresen MVP 및 마일스톤

이 문서는 **무엇을 먼저 만들지와 그 순서**를 정의한다. 시스템 동작 방식은 [DESIGN.md](DESIGN.md), 제품 정의는 [SPEC.md](SPEC.md)를 참고한다.

---

## 1. MVP 목표

Toresen의 최종 형태는 공용 호스팅 서비스(트랙=레포 포크 + 커뮤니티)지만, MVP는 그 **토대인 훈련 루프 엔진**을 먼저 세운다. 한 개의 그래픽스 강의실을 끝까지 돌릴 수 있는 자기 훈련 루프를 자기 컴퓨터(로컬호스트)에서 동작시킨다. AI가 과제를 발급하고 → 사용자가 코드를 push하면 → 코드 리뷰와 렌더 검증을 받고 → 통과하면 다음 과제로 넘어가는 사이클이 실제로 한 바퀴 이상 돈다.

성공의 기준은 화려함이 아니라 **"이 도구만으로 실제 그래픽스 과제(예: 레이캐스터)를 끝까지 해냈다"**이다. 호스팅·포크·커뮤니티 같은 상위 레이어는 이 엔진 위에 후속 마일스톤으로 얹는다(영구 제외가 아니라 순서상 뒤).

---

## 2. MVP 범위

**포함 (엔진 토대)**
- 단일 그래픽스 강의실 1개 (예: 레이캐스터 단계), 단일 사용자, 로컬호스트 실행
- 훈련 루프: 과제(Issue) → push → PR → 리뷰 → 코멘트 → 합격 게이트 → 수동 머지 → 다음 과제
- 코드 리뷰(본인 CLI) + 기계적 1차 검사(빌드·금지 기능)
- 렌더 검증(환경 고정 + 캡처 + SSIM)과 정답 자료 생성
- 리뷰어 봇(GitHub App) + GitHub 연동(Issue·PR·코멘트)
- 상태 저장과 기본 대시보드(리뷰 미러 + 진척), 레벨 보정(내부)

**후속 마일스톤으로 (영구 제외 아님)**
- 공용 호스팅 다중 사용자 운영(권한 격리·작업 큐 분리)
- 트랙=레포 **포크** 모델 (공개/포크 허용·진도 0 복제·출처 표기)
- 커뮤니티 홈(타임라인) · 둘러보기 · 프로필(학습 열정 온도·만든 강의실)
- 강의실 생성 위저드(목표·레퍼런스·초기 진단·실라버스·마무리)
- 두 기기 물리적 분리

**제외 (당분간)**
- 다중 언어
- 포크 upstream 동기화
- 커뮤니티 자유 메시지/포스팅(리액션부터 검토)
- 잠그기 어려운 과제의 멀티모달 시각 평가

> 게이미피케이션 주: 연속(streak)은 강박 유발로 **채택하지 않는다.** 대신 학습 열정 온도(프로필)·활동 기록(잔디, 연속 카운터 없음)을 쓴다.

## 3. 빌드 마일스톤

이른 단계일수록 수동·하드코딩 지름길을 허용하고, 뒤로 갈수록 자동화·상위 레이어로 확장한다.

### 마일스톤 0 — 사전 스파이크

가장 불확실한 부분을 먼저 확인한다.

- 화면 없는 컨테이너에서 GPU로 한 프레임을 렌더해 이미지 파일로 저장되는지 확인한다.
- 본인 LLM CLI를 호출해 정해진 형식(JSON)의 답을 받아오는지 확인한다.
- 빌드·렌더·캡처는 **모드 A(사용자 측, 기본)** 기준으로 스파이크한다. 모드 B(서버 측)는 같은 작업 큐 계약 위에서 작업자 위치만 바뀌는 후속 확장이다(SPEC 10장).

**완료 기준:** 두 가지가 각각 동작하는 최소 예제가 돈다.

### 마일스톤 1 — 루프 골격

GitHub·봇·상태의 뼈대를 세운다. 리뷰 품질은 아직 신경 쓰지 않는다.

- 수동으로 만든 학습용 저장소 + 리뷰어 봇(초기엔 수동 토큰 지름길, 호스팅 단계에서 GitHub App으로 — DESIGN 5.2), 하드코딩한 과제 1개로 시작한다.
- 과제를 Issue로 발급 → PR을 폴링으로 감지 → 더미 코멘트 작성 → 상태 갱신.

**완료 기준:** 더미 과제가 발급되고, PR을 올리면 봇이 코멘트를 달고 상태가 바뀐다.

### 마일스톤 2 — 코드 리뷰

실제 리뷰를 붙인다.

- 본인 CLI로 코드 리뷰 실행 → JSON 형식(DESIGN 4.2) 수신 → 코멘트로 변환(DESIGN 4.3).
- 기계적 1차 검사 → 게이트 판정(DESIGN 4.4) → 수동 머지 → 다음 과제 발급.

**완료 기준:** 코드만 있는 과제를 실제로 리뷰받고, 통과/반려가 게이트로 갈리며, 통과 시 다음 과제가 나온다.

### 마일스톤 3 — 렌더 검증

- 골든 파이프라인 [그래픽스 모듈](golden_pipelines/graphics.md)로 정답 화면을 생성·검증해 숨겨 보관한다(베이스: [COURSE_GUIDELINE_PIPELINE.md](COURSE_GUIDELINE_PIPELINE.md)). 다른 도메인 모듈은 후속.
- 스타터 템플릿 캡처 모드 + 캡처 + SSIM 비교 → 렌더 축을 게이트에 반영(DESIGN 4.5).

**완료 기준:** 화면 결과물이 있는 과제에서, 렌더가 정답과 다르면 반려되고 맞으면 통과된다.

### 마일스톤 4 — 강의실 생성·진단·대시보드·온보딩

단일 사용자 경험을 완성한다.

- 강의실 생성 흐름: 목표 → 레퍼런스 → **초기 진단(0~10 자가 체크 + 모의 과제)** → 실라버스(`syllabus.md`) → 마무리(제목·공개/포크·저장소). 하드코딩 트랙을 대체한다.
- 온보딩 분산: 가입 셋업 마법사(환영·GitHub·트레이너), AI 연결=설정+호출 시점 경고, 레포 등록=생성 마무리.
- 대시보드(스킬 트리 진척, 리뷰 미러, 트레이너 페르소나) + 레벨 보정(내부, 비노출).

**완료 기준:** 처음 켜서 셋업하고, 직접 정한 주제로 강의실을 만들어 루프를 돌릴 수 있다.

### 마일스톤 5 — 호스팅·커뮤니티·포크 (상위 레이어)

엔진 위에 공용 서비스 계층을 얹는다.

- 다중 사용자 호스팅(권한 격리·강의실 데이터 접근 제어).
- 트랙=레포 포크(공개/포크 허용·진도 0 복제·출처 표기·독립 스냅샷).
- 커뮤니티 홈(옵트인·성취 중심) · 둘러보기 · 프로필(학습 열정 온도·만든 강의실).
- **연산 모드 B(서버 중앙처리) 옵션** — 공개 신뢰 신호(통과자 수 등)를 보증하려는 운영자를 위해 빌드·렌더·유사도 비교(필요 시 LLM 추론까지)를 서비스 작업자로 돌린다. 같은 작업 큐 계약 위에서 작업자 위치만 바꾼다. 모드 A에서 통과 기록은 자가 보고이므로 공개 보증 신호로 쓰지 않는다(SPEC 4·10장).

**완료 기준:** 다른 사용자의 공개 강의실을 둘러보고 포크해 내 강의실로 시작할 수 있다. 모드 B 배포에서는 통과자 수가 서버 검증으로 보증된다.

---

## 4. 성공 기준

MVP가 끝났다고 보는 시점은 **사용자가 이 도구만으로 그래픽스 강의실 하나(예: 레이캐스터)를 처음부터 끝까지 완주**하고, 그 과정에서 리뷰·렌더 검증·게이트가 실제로 학습에 도움이 되었을 때다. 그 위에 호스팅·커뮤니티·포크(마일스톤 5)를 얹어 공용 서비스로 확장한다.
