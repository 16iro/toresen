# 골든 파이프라인 — 그래픽스 모듈

이 문서는 [코스 가이드라인 파이프라인(베이스)](../COURSE_GUIDELINE_PIPELINE.md)을 **그래픽스 도메인**에 맞게 구현한 모듈이다. 골든의 정의·공통 흐름·은닉·포크 규칙·검증 층의 일반 형태는 베이스를 따르고, 여기서는 그래픽스 특유의 자료 형태·생성법·비교법만 정의한다. (전체 구조는 [DESIGN.md](../DESIGN.md), 제품 정의는 [SPEC.md](../SPEC.md) 참고.)

이 모듈은 첫 시드 도메인이며, MVP의 렌더 검증(DESIGN 4.5)이 이 모듈을 기준으로 동작한다.

---

## 1. 골든 자료의 형태

- **정답 화면** — 단계별 고정 환경에서 정답 구현을 렌더한 기준 이미지. 채점 비교의 기준이 된다. 모드 B에서는 채점 서버가 정답 구현을 제출물과 같은 환경에서 렌더해 기준 화면을 얻으므로 환경 차이가 없다(한 번 렌더해 캐시, DESIGN 4.5). 생성·검증 시 보관한 화면들은 허용 범위가 된다.
- **허용 범위(envelope)** — 올바른 구현끼리도 음영·필터링·부동소수 차이로 미세하게 갈릴 수 있으므로(DESIGN 4.5), 정답 화면은 하나가 아니라 여러 개를 허용 범위로 둘 수 있다.
- **정답 구현** — 위 화면을 만든 구현. 숨겨 보관하며 학습자에게 절대 노출하지 않는다.

## 2. 입력

- 학습 계약서(단계별 범위·합격 기준), 참고 자료(원본 코드·논문·강의 자료).
- **단계별 고정 환경** — 모델·텍스처·카메라·조명·해상도와, 채점 대상이 아닌 렌더 설정(색공간·텍스처 필터링·안티앨리어싱 등). 이 환경 정의는 학습자 스타터 템플릿·캡처와 **동일한 하나의 정의를 공유**한다(어긋나면 채점 비교가 무의미하다).

## 3. 정답 생성

- **참고 자료 기반 작성** — 기억이 아니라 강의실의 참고 자료(원본 알고리즘·논문)를 근거로 작성한다.
- **충분한 연산 투입** — 한 번만 도는 작업이므로 가장 강한 모델과 충분한 추론을 사용한다(원작자 LLM에서 실행, BYO).
- **빌드·렌더 자가 수정** — 작성 → 빌드 → 실행·렌더 → 오류 확인 → 수정을 반복한다.
- **단계별 점진 작성** — 마일스톤 단위로 작게 만든다.
- **기존 구현 우선** — 신뢰할 수 있는 기존 구현이 있으면 범위에 맞춰 활용한다.

## 4. 비교 방법 (그래픽스 특화)

- **3구간 판정** — 캡처를 정답 화면과 구조적 유사도(SSIM 등)로 비교해, 높음(자동 통과)·명백한 불일치(자동 반려)·중간(멀티모달·본인 확인)으로 나눈다. 단일 임계값 이분법을 쓰지 않는다(DESIGN 4.5, SPEC 6.3).
- **구조 특징 비교** — 전체 프레임 SSIM이 음영·필터링 차이에 취약한 과제는 엣지·영역별 레이아웃·색 분포 같은 구조 특징 비교를 함께 쓴다.
- **허용 범위 매칭** — 학습자 화면이 envelope의 어느 기준 화면에든 부합하면 통과로 본다.

## 5. 검증 (베이스 검증 층의 그래픽스 구현)

- **반복 생성 일치** — 정답을 여러 번 독립 생성해 화면이 일치하는지 본다. 무해한 차이는 envelope로 보관, 구조적으로 갈리면 사람 확인.
- **원본 대조** — 알려진 원본(유명 게임 등)이 있으면 해당 범위에 한해 비교한다.
- **수학적 불변식** — 결과가 만족할 성질(대칭, 거리-크기 비례 등)을 자동 검사한다.
- **기본 점검** — 빈 화면이 아님, 기대한 형태가 있음, 색 분포가 정상 범위임을 자동 확인한다.
- **교차 검토** — 별도 검토로 정답 구현의 논리 오류를 점검한다.
- **골든 자가 검증(불변식)** — 노드 공개 전에 정답 구현을 그 노드의 렌더 게이트에 직접 통과시켜, 정답 화면이 자기 3구간 기준에서 "높음"으로 떨어지는지 확인한다. 통과 못 하면 그 노드는 출시·포크되지 않는다(COURSE 베이스 5장).
- **최종 사람 확인** — 통과한 정답 화면을 원작자가 눈으로 확인한다. 화면만 보므로 학습에 영향을 주지 않는다.
