# 코스 가이드라인 파이프라인 (베이스)

이 문서는 한 학습 강의실(트랙)의 **채점 기준이 되는 정답 자료(골든)를 만들고 검증하는 과정**의 **공통 골격**을 정의한다. 골든의 구체적 형태(정답 화면 / 테스트 묶음 / 기대 입출력 …)와 비교 방법은 도메인마다 다르므로, 도메인별 **모듈**로 분리해 채운다. 이 베이스는 모든 모듈이 공유하는 정의·흐름·규칙과, 모듈이 채워야 할 인터페이스를 정한다.

(전체 시스템 구조는 [DESIGN.md](DESIGN.md), 제품 정의는 [SPEC.md](SPEC.md) 참고. 이 파이프라인은 제출물을 채점하는 실행 흐름과 별개로 강의실 생성 시 정답 자료를 만드는 과정이다 — DESIGN 2.3.)

---

## 1. 목적

과제를 자동으로 채점하려면 단계마다 **무엇과 비교할 정답 자료(골든)**가 있어야 한다. 강의실은 개인이 생성하므로(SPEC 4장), 이 파이프라인은 **원작자가 강의실을 만들 때** 학습 계약서(syllabus)가 정해진 뒤 **단계별 정답 구현과 골든 자료를 만들고 검증해 숨겨 보관**한다. 학습자는 정답 구현을 절대 보지 못하며, 자신의 코드는 100% 직접 작성한다.

여기서 만든 골든이 그 강의실의 채점 기준, 곧 **"이 강의실 기준"**이 된다(공인된 표준이 아니라 원작자가 정한 값). 강의실을 **포크**하면 골든도 강의실 정의의 일부로 함께 복제되어, 포크한 학습자도 동일한 기준으로 채점받는다.

이 과정은 강의실을 만들 때 한 번 실행되고 자주 돌지 않는다. 정확성이 가장 중요하므로 비용이 큰 기법도 적극 사용한다(원작자 LLM에서 실행, BYO).

---

## 2. 골든은 도메인마다 형태가 다르다

골든은 "정답 판정 자료"의 총칭이며, 도메인에 따라 구체 형태와 비교 방법이 달라진다.

| 도메인 | 골든 자료 형태 | 비교 방법 |
|---|---|---|
| 그래픽스 | 정답 화면(+허용 범위) + 정답 구현 | 구조적 유사도 3구간(SSIM 등) |
| 알고리즘 / CLI | 정답 구현으로 뽑은 (입력→기대출력) 쌍 + 불변식 + 숨긴 테스트 묶음 | 정확 일치 / 속성(불변식) 검사 |
| 웹 백엔드 | (요청→기대응답) 계약 테스트 + 상태코드·스키마 검사 | 계약·스키마 일치 |

골든 자료 형태와 비교 방법은 **강의실 파이프라인 설정의 일부**로 저장된다(DESIGN 2.3). 그래픽스는 그중 한 설정일 뿐이다.

---

## 3. 공통 흐름

```
학습 계약서 확정 (강의실 생성)
   │
   ▼
단계마다:  정답 생성 ──▶ 정답 검증 ──▶ 통과하면 골든을 숨겨 보관
                                  └ 실패하면 다시 생성
```

생성과 검증은 각각 **여러 독립 기법의 묶음**이며, 새 기법을 더하거나 빼도 나머지에 영향이 없도록 구성한다.

---

## 4. 정답 생성 (도메인 무관)

올바른 정답 구현을 만들기 위해 다음을 조합한다. 산출물의 형태(화면·출력·응답)는 모듈이 정하지만, 생성 전략은 도메인과 무관하다.

- **참고 자료 기반 작성** — 기억이 아니라 강의실의 참고 자료(원본 알고리즘·논문)를 근거로 작성한다.
- **충분한 연산 투입** — 한 번만 도는 작업이므로 가장 강한 모델과 충분한 추론을 사용한다.
- **빌드·실행 자가 수정** — 작성 → 빌드 → 실행 → 결과 확인 → 수정을 반복한다(그래픽스면 렌더, 알고리즘이면 테스트 실행).
- **단계별 점진 작성** — 마일스톤 단위로 작게 만든다.
- **기존 구현 우선** — 신뢰할 수 있는 기존 구현이 있으면 범위에 맞춰 활용한다.

---

## 5. 정답 검증 (도메인 무관 검증 층)

생성된 골든이 실제로 옳은지 여러 층으로 확인한다. 각 검증은 독립적이며 통과 기준·항목은 조정할 수 있다. 아래 층은 일반 형태이고, 모듈이 도메인에 맞게 구현한다.

- **반복 생성 일치** — 정답을 여러 번 독립 생성해 결과가 서로 일치하는지 본다. 무해한 차이로 갈리면 **허용 범위로 함께 보관**하고, 구조적으로 갈리면 신뢰할 수 없으므로 사람 확인으로 넘긴다.
- **원본 대조** — 알려진 원본이 있으면 해당 범위에 한해 비교한다.
- **불변식 검사** — 결과가 만족해야 할 성질을 자동 검사한다(그래픽스: 대칭·비례 / 알고리즘: 정렬됨·순열임·왕복 등).
- **기본 점검** — 자명하지 않은 출력인가, 기대한 형태가 있는가를 자동 확인한다(그래픽스: 빈 화면 아님·색 분포 / 그 외: 비어있지 않음·형식 적합).
- **교차 검토** — 별도 검토 과정으로 정답 구현의 논리 오류를 점검한다.
- **골든 자가 검증(불변식)** — 노드를 공개하기 전에 **그 노드의 정답 구현이 그 노드의 게이트(면역 바닥, DESIGN 4.4)를 실제로 통과하는지** 자동 확인한다. "골든은 자기 기준을 통과한다"가 불변식이며, 통과하지 못하면 그 노드는 출시·포크되지 않는다. 풀 수 없는·틀린 골든이 학습자에게 도달하기 전에 걸러진다.
- **최종 사람 확인** — 위를 통과한 골든을 원작자가 한 번 확인한다. 정답 구현 코드가 아니라 산출물(화면·기대출력·테스트)만 보므로 학습에 영향을 주지 않는다.

> 품질 주: 개인 생성 강의실은 골든 품질이 들쭉날쭉할 수 있다. 검증을 거친 강의실일수록 포크·통과 신호(SPEC 4장)로 신뢰가 쌓인다.

---

## 6. 공개 명세 vs 숨긴 채점 골든

골든 중 일부는 학습자에게 보여도 되고 일부는 숨겨야 한다.

- **공개 명세** — 과제의 계약·기대를 정의하는 자료(예: 고정 환경 정의, 일부 예시 테스트). 학습자가 보고 이를 충족하도록 구현해도 된다.
- **숨긴 채점 골든** — 게이밍을 막기 위해 숨기는 자료(정답 화면·정답 구현·숨긴 채점 테스트). 그래픽스의 정답 화면을 숨기는 것과 같은 원리다.

이 구분은 모듈이 도메인에 맞게 정하되, 숨긴 골든은 서비스 측에만 보관하고 학습자 저장소·클라이언트에 내려보내지 않는다(DESIGN 3.3, 연산 모드별 은닉 한계는 SPEC 10장).

---

## 7. 도메인 모듈

베이스를 구현한 도메인별 모듈은 다음과 같다. 각 모듈은 2장의 (골든 형태·비교 방법), 5장의 검증 층 구현, 6장의 공개/숨김 구분을 채운다.

- **그래픽스** (MVP, 활성) — [golden_pipelines/graphics.md](golden_pipelines/graphics.md)
- **알고리즘 / CLI** (계획) — 정답 구현 기반 (입력→기대출력) + 불변식 + 숨긴 테스트.
- **웹 백엔드** (계획) — 계약/통합 테스트 + 스키마·상태코드 검사.

---

## 8. 유연성 원칙

- 생성·검증의 각 기법은 **독립 단계**로 두어, 더하거나 빼거나 순서를 바꿔도 나머지가 영향받지 않게 한다.
- 검증의 통과 기준값(유사도 임계·허용 오차 등)은 **설정으로 분리**해, 코드를 고치지 않고도 조정한다. 이 기준이 곧 그 강의실의 채점 기준이다.
- 한 번 통과해 고정된 골든은 그대로 보관하며, 과제 범위가 바뀔 때만 다시 만든다.
- 새 도메인 지원은 베이스를 고치지 않고 **새 모듈을 추가**하는 것으로 끝나도록 한다.
