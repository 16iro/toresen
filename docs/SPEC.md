# Toresen 기획서

## 1. 개요

Toresen은 LLM을 **엄격한 시니어 리뷰어**로 활용하는 자기주도 하드코어 트레이닝 시스템이다.

가장 중요한 원칙은 **AI가 코드를 대신 작성하지 않는다**는 것이다. AI는 학습자의 코드를 리뷰하고, 다음 과제를 출제하고, 개념을 설명한다. 그러나 구현 코드나 복사해 쓸 수 있는 템플릿은 제공하지 않는다. 학습자는 스스로 코드를 작성하며 자신의 한계를 넘어선다.

기존 코딩 학습 도구가 "막히면 답을 보여주는" 방향이라면, Toresen은 "막혀도 스스로 풀게 하되, 시니어가 옆에서 매섭게 피드백하는" 도장(dojo)에 가깝다.

Toresen은 **공용 호스팅 서비스**로 운영된다. 다수의 사용자가 각자 **강의실(트랙)을 직접 생성**하고, 공개된 강의실을 서로 **둘러보고 포크(fork)**해 자신의 학습으로 가져온다. 무거운 연산(LLM 추론·빌드·렌더)의 실행 위치는 **연산 모드**로 정한다(10장) — 기본값인 **모드 A**에서는 각 사용자의 환경에서 실행되고 비용도 본인이 부담하며(BYO), **모드 B**에서는 운영자가 서버에서 수행해 공개 신뢰 신호를 보증한다. 어느 쪽이든 서비스는 조율·커뮤니티·강의실 정의·상태를 호스팅한다.

## 2. 범위

강의실은 개인이 생성하므로, 도메인과 언어는 **각 강의실이 정한다**. 서비스 차원에서 단일 도메인으로 제한하지 않는다. 첫 시드/예시 강의실은 **그래픽스 프로그래밍 훈련**을 다루며, 학습자는 3D 그래픽스의 역사적 이정표를 직접 클론하며 단계적으로 올라간다 — 소프트웨어 레이캐스팅(울펜슈타인) → 공간 분할과 2.5D(둠) → 순수 3D와 셰이더 → Vulkan 기반 모던 렌더링 엔진. 알고리즘·자료구조, 웹 백엔드 등 다른 주제의 강의실도 동일한 구조로 생성된다.

| 구분 | 내용 |
|---|---|
| 도메인 | 강의실별로 개방 (첫 시드 예시는 그래픽스) |
| 언어 | 강의실별로 지정 (예: C++ 또는 Rust) |
| 대상 학습자 | Git과 터미널에 익숙한 중급 이상 개발자 |
| 연산 위치·비용 | 연산 모드로 정함 — 모드 A(기본)는 본인 부담 BYO, 모드 B는 운영자 부담 (10장 참고) |

## 3. 핵심 원칙

1. **AI 코드 생성 금지.** 리뷰·과제 출제·개념 설명만 한다. 아키텍처 원칙이나 트레이드오프 설명은 허용하지만, 완성된 함수 본문이나 복사 가능한 구현 블록은 제공하지 않는다.
2. **연산 모드 (A 기본 / B 선택).** 무거운 연산(LLM 추론, 빌드·렌더·유사도 비교)을 어디서 수행할지는 배포 단위로 정하는 설정이다. **모드 A**(기본)에서는 거의 모든 연산이 사용자 본인의 LLM CLI·환경에서 실행되고 비용도 본인이 부담한다(BYO. 단 골든과의 유사도 비교만은 골든을 숨기기 위해 서버에서 — 10장). **모드 B**에서는 운영자가 이 연산을 서비스 측에서 수행하고 비용을 부담한다. 두 모드는 같은 작업 큐 계약 위에서 전환되며, 화면·상태·강의실 정의는 모드와 무관하게 동일하다(10장).
3. **호스팅 조율 계층.** 서비스는 강의실 정의·상태·커뮤니티·프로필·대시보드·GitHub 연동을 호스팅한다. 무거운 연산의 실행 위치는 연산 모드(2번·10장)를 따른다.
4. **에이전트가 저장소를 직접 읽는다.** 리뷰 시 변경분만 보는 것이 아니라, 리뷰 엔진이 저장소 전체를 읽어 호출 관계·의존성·빌드 결과까지 직접 확인한다.

## 4. 제품 모델: 강의실 = 레포 (포크)

강의실(트랙)은 GitHub 레포의 멘탈 모델을 따른다.

- **개인 생성.** 강의실은 운영 주체가 카탈로그로 미리 만들어 두는 것이 아니라, **개인이 자신의 수요에 맞춰 생성**한다(5장의 생성 흐름). 레퍼런스·실라버스·과제 내용이 그 개인의 목표에 맞게 결정되고, 결과는 서버에 모델 데이터로 저장된다.
- **공개와 포크.** 강의실은 공개/비공개로 둘 수 있고, **공개일 때만 포크를 허용**할지 별도로 정한다(공개와 포크 허용은 별개 축). 다른 사용자는 공개 강의실을 둘러보고, 포크 허용 시 자신의 강의실로 복제한다.
- **포크 규칙 (MVP).**
  - 포크는 **강의실 정의(커리큘럼·과제 정의·파이프라인 설정·골든·레퍼런스 자산)를 스냅샷으로 복제**한다. 제3자 레퍼런스 원본 소스는 라이선스상 복제하지 않고 출처 인용만 가져온다.
  - **진도는 0부터 시작**하며, 원작자의 통과 기록·리뷰·활동은 가져오지 않는다.
  - 포크는 **독립 스냅샷**이다. 원본의 이후 변경을 따라가는 upstream 동기화는 하지 않는다(향후 확장).
  - **출처(원작자)를 표기**한다("○○에서 포크함"). 포크의 포크도 허용하되 출처 표기는 유지한다.
  - 포크는 정의 복제로 끝나지 않고 **포크한 사람의 학습 레포를 새로 마련**해야 작동한다. 실행 절차와 충돌 처리(레포·봇·syllabus 프로비저닝, 라이선스, 골든 환경 결합, 계보 집계)는 [FORK.md](FORK.md)에서 정의한다.
- **유사도 기준은 "이 강의실 기준".** 결과물 검증의 정답·유사도 기준(6장)은 공인된 표준이 아니라 **원작자가 정한 값**이다. 화면에서도 "이 강의실 기준"임을 인지시킨다.
- **품질 신호.** 개인 생성 강의실은 품질이 들쭉날쭉할 수 있으므로, **포크 수·통과자 수·원작자**를 신뢰 신호로 노출한다(GitHub의 star/fork에 해당). "통과 N명"은 원본과 그 포크 계보를 합산한 수다. 이 집계는 **PR 머지도 LLM 판정도 아니라 면역 바닥(빌드·테스트·렌더 구조 비교)의 "검증된 통과"**를 센다 — 학습자가 자기 레포를 직접 머지(또는 강제 통과)하거나 리뷰어를 인젝션해 지적을 지워도 그것만으로는 집계되지 않으므로, 레포 소유권을 학습자가 가진 채로도 위조되지 않는다(6.4·DESIGN 4.4·4.8). 검증된 통과 집계는 연산 모드 B에서 성립하며(10장), 모드 A에서는 바닥 검사도 자가 보고이므로 통과를 개인 진척으로만 쓴다.

## 5. 강의실 생성과 동작 흐름

### 5.1 동작 흐름

```
강의실 생성 (위저드)
   목표 → 레퍼런스 → 초기 진단 → 실라버스 → 마무리(제목·공개·포크·저장소)
   │
   ▼  syllabus.md (학습 계약서) 확정 + 학습용 저장소 연결
   │
   ▼
┌─────────────── 훈련 루프 (반복) ───────────────┐
│  과제 발급(Issue) → 코딩 → Pull Request 제출   │
│        → 리뷰 → 통과 시 다음 과제              │
│        → 반려 시 이전 피드백 유지하며 재시도   │
│        → 교착 시 강제 통과(override) 비상구    │
└────────────────────────────────────────────────┘
```

역할 분담은 다음과 같다.

- **학습자(사람):** 강의실 생성(목표·레퍼런스 제시), 코드 작성, 렌더링 화면 직접 확인, 리뷰에 대한 비판적 수용과 반론.
- **AI:** 커리큘럼(실라버스) 제안, 다음 과제 생성, 코드 리뷰, 통과/반려 판정.

### 5.2 생성 흐름은 위저드로

강의실 생성은 5단계 위저드로 진행한다.

1. **목표** — 무엇을 만들고/배우고 싶은지 자유 입력.
2. **레퍼런스** — 참고 자료(책·논문·URL·레포) 추가(5.3절).
3. **초기 진단** — 학습자의 출발 수준을 잡는다(5.5절).
4. **실라버스** — 트레이너가 목표·레퍼런스·진단 결과를 근거로 커리큘럼(스킬 트리 → 노드별 과제 명세)을 제안하고, 학습자가 편집한다. 생성은 LLM 1회 호출이다. 여기서 확정된 **노드별 과제 명세(요구사항)는 고정**되며, 골든 파이프라인이 이 명세에 맞춰 정답 자료를 만든다(COURSE). 이후 훈련 루프에서 발급되는 과제는 이 고정 명세를 학습자 스캐폴딩에 맞춰 제시하는 것이지 요구사항을 새로 짓는 것이 아니다(5.6).
5. **마무리** — 제목(트레이너 제안 + 수정)·소개, 공개 설정(공개 → 포크 허용 토글), 학습용 저장소 연결(10장). **난이도를 사용자가 고르는 설정값은 두지 않는다**(난이도는 진단·성과로 정해지는 값이지 생성자가 정하는 계수가 아니다).

### 5.3 레퍼런스 기반 커리큘럼 생성

LLM의 내부 지식에만 의존하면 일반론적이거나 학습자 의도와 다른 커리큘럼이 나온다. 따라서 **기준 문서(reference)를 근거로** 커리큘럼을 구성한다. 세 가지 소스를 지원하며, 각각 신뢰도가 다르다.

| 소스 | 처리 방식 | 신뢰도 |
|---|---|---|
| 코드베이스 (예: 둠 오픈소스) | 모듈 구조와 의존 관계를 추출하되, 인터페이스와 역할만 사용한다. 실제 구현 코드는 학습자에게 노출하지 않는다(정답지 역할). | 높음 |
| 논문 (예: SIGGRAPH 자료) | 핵심 알고리즘과 함께, 논문이 전제하는 선수 지식까지 학습 항목으로 끌어낸다. | 중간 |
| 강의 슬라이드 (예: GDC/CEDEC) | 슬라이드 이미지를 멀티모달로 읽고, 가능하면 발표 트랜스크립트를 함께 사용한다. 정보 공백이 많다. | 낮음 |

생성된 모든 학습 항목에는 **출처와 신뢰도를 함께 표시**한다. 학습자는 최종적으로 이 항목들을 검토하고 가지치기한다.

### 5.4 전체 지형 제시 후 범위 합의

학습자에게 범위를 묻기 전에, 시스템이 **해당 주제의 전체 기술 지형을 먼저 제시**한다. 예를 들어 울펜슈타인이라면 "핵심 기술은 레이캐스팅·텍스처 매핑·스프라이트·맵 포맷·입력·오디오"라고 펼쳐 보인다. 학습자는 이 지형을 본 뒤에 범위를 정한다("레이캐스팅과 텍스처까지만, 오디오는 단순화"). 모르는 상태에서 찍는 것이 아니라, 알고 선택하게 한다.

이때 **각 항목의 정량적 난이도(보면상수)는 사용자에게 노출하지 않는다**(5.6절). 지형 제시는 "무엇이 있는지"를 보여주는 것이지 "각 항목이 몇 점짜리인지"를 보여주는 것이 아니다.

### 5.5 초기 진단 — 자가 체크 + 모의 과제 검증

초기 진단은 학습자의 **출발 수준**을 잡기 위한 것이며, 점수를 매기는 시험이 아니다. 두 토막으로 구성한다.

1. **자가 체크.** 학습자가 "이 주제를 얼마나 아는지"를 **0~10 슬라이더 한 개로 스스로 신고**한다(시스템이 매기는 점수가 아니라 사용자 자가 신고). 슬라이더 값에 정성적 라벨(처음이에요 … 아주 능숙해요)을 붙인다.
2. **모의 과제로 검증.** 주제에서 뽑은 가벼운 인앱 과제(인라인 코드 에디터)를 풀게 해 자가 신고를 검증·보정한다. 제출 시 연결된 LLM이 1회 평가한다. 건너뛰면 거친 기본값에서 시작한다. 진단은 적응형이라 수행에 따라 과제가 한두 개 더 이어질 수 있다.

진단 결과는 실라버스 생성의 입력이 되고, 이후 매 과제 결과로 동적으로 보정된다(8장·5.6절). 진단은 화면상 "출발점을 맞춘다"로만 안내하고, 등급·점수·구간을 결과로 노출하지 않는다.

> 주: 본 절은 기존 "초/중/고급 거친 설문" 방식을 대체한다. "정밀한 단일 점수를 시스템이 매겨 신뢰·노출하지 않는다"는 원칙은 유지하되, **사용자가 스스로 입력하는 0~10 자가 신고 + 과제 검증**으로 정밀화한 것이다.

### 5.6 난이도의 다룸과 은닉

난이도는 **잴 수 있는 점수가 아니라 학습 경로 위의 위치**로 다룬다. 정교한 난이도 수치를 시스템이 알아내려 하지 않는다.

- **위치는 스킬 트리가 정한다.** "어디까지 왔는가"는 작가가 짠 스킬 트리(5.7) 위의 위치이지 0~10 스칼라가 아니다. 각 노드에는 작가가 매긴 거친 순서 티어("보면상수")가 붙되, 이는 공인된 객관값이 아니라 **"이 강의실 기준"의 상대 순서**다. 통과 데이터가 쌓이면 경험적으로 보정될 수 있으나, MVP·신규 강의실에서는 작가가 정한 순서만으로 충분하다.
- **적응은 점수 조절이 아니라 도움(스캐폴딩) 조절이다.** 같은 경로 위에서, 학습자가 수월하면 다음 과제의 도움을 줄이고(간결하게·한 덩어리), 헤매면 도움을 늘린다(잘게 분할·힌트·**보조 셋업 코드 더 제공**). 도움은 채점 대상이 아닌 보조만 건드린다 — 그래픽스의 채점용 렌더 환경(6.3의 단일 출처)은 스캐폴딩으로 바뀌지 않는다. 과제 발급에는 정체불명의 난이도 숫자가 아니라 **(노드의 고정 명세 + 도움의 양)**을 쓴다(8장·DESIGN 3.5). **과제 요구사항 자체는 노드에 고정**되어 골든과 짝을 이루고(5.7), 도움 양은 그 요구사항을 **어떻게 제시하느냐**만 바꾼다 — 그래서 적응이 채점 기준을 흔들지 않는다.

이 모든 내부 값(노드 티어·도움의 양·레벨 추정치)은 **사용자에게 철저히 숨긴다.** 다음을 화면에 표시하지 않는다.

- 과제·강의실의 정량적 난이도 숫자(0~10)나 구간.
- "더 어려운/쉬운 과제로 넘어간다" 같은 상대적 난이도 변화.
- 학습자에 대한 정밀 점수.

이유: 난이도가 보이면 학습자가 자기를 숫자에 가두고(자기 수준 역산), 진단을 "높게 받아야 하는 시험"으로 인식하며, 일부러 틀려 쉬운 트랙을 받는 등의 게이밍이 생긴다. 학습 출발점을 맞추려는 도구가 자기 평가 점수표로 변질되는 것을 막는다. 강의실 수준을 굳이 보여야 하는 탐색 화면에서도 "입문용/심화" 같은 거친 정성 라벨까지만 허용한다.

### 5.7 스킬 트리와 학습 계약서

합의 결과는 단순 목록이 아니라 **의존성 그래프(스킬 트리)**로 표현한다. 선행 개념이 후행 과제의 전제가 되도록 순서를 잡는다(예: 벡터 수학 → 공간 분할 순회). 이 그래프는 `syllabus.md`로 저장소에 고정되며, 이후 모든 리뷰가 참조하는 **학습 계약서**가 된다.

## 6. 리뷰 시스템

### 6.1 리뷰 입력

리뷰 엔진은 변경된 코드 조각만 보지 않는다. 저장소 전체에 접근하여 **변경된 부분 밖의 맥락**(호출하는 쪽, 의존하는 파일, 빌드 결과)까지 직접 확인한다. 아키텍처 리뷰는 본질적으로 변경 파일 바깥의 관계를 봐야 하기 때문이다. 나아가 강의실의 참고 코드베이스를 미리 구조화한 **레퍼런스 자산**(코드 리뷰 파이프라인)을 근거로 "레퍼런스는 이 문제를 어떻게 다루는가"까지 참조한다.

### 6.2 코드 분석 (주 평가축)

평가의 중심은 코드 분석이다. 아키텍처와 설계 — 모듈 분리, 의존성 방향, 메모리 레이아웃, 언어 관용구, 금지된 API 사용 여부 — 를 검증한다. 이 중 명확히 판정 가능한 항목(컴파일 성공 여부, 금지 API 사용, 테스트 통과)은 LLM이 추측하지 않고 **컴파일러·정적 분석 도구·테스트가 1차로 결정**한다.

### 6.3 렌더링 검증 (보조 평가축)

그래픽스는 코드가 옳아 보여도 실제 화면이 잘못될 수 있다. 이를 잡기 위해 렌더링 결과를 검증한다.

- **위치:** 합격은 가중 점수 합산이 아니다(점수는 표시용, DESIGN 4.4). 렌더 구조 비교는 **면역 바닥**의 일부로 통과를 좌우하고(시각 과제), 코드 분석은 그 위 학습용 피드백 층이다. 화면 결과가 없는 과제는 빌드·테스트 바닥 + 코드 분석으로 본다. 렌더는 **증거이지 단독 심판이 아니다** — 단독 차단은 화면이 명백히 깨진 경우뿐이고, 애매한 차이는 코드 지적과 함께일 때만 무게를 싣는다. (피드백상 비중은 대략 코드 분석이 크고 렌더가 보조이나, 이는 가중 합격 점수가 아니다.)
- **입력 고정(soft-lock):** 과제는 렌더링에 필요한 환경(모델·텍스처·카메라·조명·해상도)과 채점 대상이 아닌 렌더 설정(색공간·텍스처 필터링·안티앨리어싱 등)을 스타터 템플릿 수준에서 고정한다(가이드라인·뼈대일 뿐 구현 코드를 주는 것은 아니다). 환경 고정은 기대 출력의 **범위를 좁히는 것**이지 단 하나로 못박는 것이 아니다 — 같은 알고리즘도 부동소수 누적 순서·드라이버 차이 등으로 미세하게 갈릴 수 있기 때문이다.
- **3구간 판정:** 고정된 환경에서 화면을 캡처하고 정답 이미지와 **구조적 유사도(SSIM 등)로 비교**해 세 구간으로 나눈다 — **높음**(기준 이상)은 자동 통과, **명백한 불일치**(기준에 크게 못 미침)는 자동 반려, **중간 구간**은 멀티모달 판정이나 학습자 본인 확인으로 넘긴다. 마진이 살짝 모자란 것만으로 하드 반려하지 않는다. 단일 임계값으로 통과/반려를 딱 가르지 않는다. 비교 기준·구간값은 원작자가 정한 **"이 강의실 기준"**이다(4장).
- **허용 범위(envelope):** 정답 생성에서 독립 구현들이 미세하게 갈리면(COURSE 5장 반복 생성 일치) 하나로 강제하지 않고 **여러 기준 화면을 허용 범위로 보관**해, 학습자 화면이 그중 하나에 부합하면 통과로 본다.
- **견고한 비교:** 전체 프레임 SSIM이 음영·필터링 차이에 취약하므로, 필요한 과제는 **구조 특징(엣지·영역별 레이아웃·색 분포)** 기반 비교를 함께 써서 무해한 차이엔 관대하되 구조 오류(잘못된 위치·누락)는 잡는다.
- **멀티모달 보조:** 환경 고정이 어려운 과제, 중간 구간 판정, "무엇이 잘못되었는지" 설명이 필요한 경우에 멀티모달 판정을 보조로 쓴다.

### 6.4 리뷰 신뢰성

- **근거 강제(기계 검증).** 모든 지적은 실제 코드 위치(파일·라인)와 그 위치의 코드 인용을 달아야 한다. 앱은 게시 전에 인용이 그 위치에 **실제로 존재하는지 기계로 대조**하고, 실패한 지적은 게시하지 않으며 차단 근거로도 쓰지 않는다. "근거를 가리킬 수 없으면 지적할 수 없다"가 부탁이 아니라 강제된다.
- **태도 분리:** 취향과 설계 원칙에는 단호하게, 확인하지 못한 사실에는 신중하게 표현한다.
- **신뢰의 바닥은 속일 수 없는 검사.** 공개로 집계되는 "검증된 통과"는 빌드·테스트·렌더 구조 비교 같은 **기계·서버 계산(인젝션 면역)**에만 걸고, LLM 리뷰는 그 위에 얹는 학습용 추가 층이다. 그래서 리뷰어를 속여 지적을 지워도 깨진 코드는 바닥에서 걸려 통과를 못 만든다(DESIGN 4.4·4.8).
- **차단은 좁게, 사람이 반박 가능하게.** LLM 지적은 기본이 "변경 요청"이고, 진행을 하드 차단하려면 면역 바닥에 걸리거나 **독립 재검증**을 통과해야 한다. 자가 신고 확신값(confidence)은 표시용일 뿐 차단 근거가 아니다. 차단에 동의하지 않는 학습자는 이의제기를 걸고 — 이는 같은 리뷰어 재탕이 아니라 **새 인스턴스 여러 개가 "이 지적을 반박하라"로 독립 실행되어 다수가 차단이라야 유지**된다(유지·강등·철회). 그래도 풀리지 않으면 강제 통과(6.6)로 빠져나온다. 기계가 코드를 재판하는 게 아니라, 근거 없는 지적이 사람을 영영 막지 못하게 하는 구조다.
- **리뷰어 인젝션 방어:** 리뷰 대상(코드·주석·커밋·캡처)은 신뢰 못 할 입력이므로 데이터로 격리하고, 리뷰어를 겨냥한 지시를 게시 전 스캔·정화한다(DESIGN 4.8).
- **트레이너 말투:** 트레이너의 말투는 프리셋(단호·차분·위트)으로 고르고 편집할 수 있으나, **말투만 달라질 뿐 지적 내용·근거·코드 인용은 동일**하다(진지함 가드).
- **코드 누출 차단:** 리뷰 본문에 복사 가능한 구현 코드(완성 함수·여러 줄 코드 블록)가 섞이면 게시 전에 걸러, "AI 코드 생성 금지"(3장)를 출력 경로에서 강제한다.

### 6.5 리뷰 일관성

훈련 루프가 길어져도 흔들리지 않도록, 매 리뷰마다 **학습 계약서(`syllabus.md`), 직전 리뷰 요약, 미해결 항목**을 함께 참조한다. 히스토리가 길어지면 마일스톤 단위로 요약해 압축한다.

### 6.6 강제 통과 처리

강제 통과(override)는 이의제기(6.4)로도 풀리지 않을 때 쓰는 **드문 최후 비상구**다. override된 제출물은 미완성 상태로 통과되며, 이후 리뷰의 참고 맥락에 섞이면 잘못된 코드가 모범으로 학습될 수 있으므로 **이후 리뷰의 참고 대상에서 제외**하고 사유를 기록한다. override 사용률이 높아지면 정상 워크플로가 아니라 리뷰 품질 경보로 본다.

## 7. 커뮤니티와 발견

Toresen이 공용 서비스이므로, 강의실을 공유·발견하는 계층을 둔다.

### 7.1 홈 — 커뮤니티 타임라인

로그인 착지 화면은 커뮤니티 활동 피드다. 다른 사용자가 과제를 통과하거나 트랙을 수료한 활동을 GitHub 홈 피드 문법(행위자 + 동사 + 임베드 카드)으로 보여준다. 안전 가드:

- **성취 중심.** 통과·수료·활동만 보여주고, **속도·등수·리더보드 등 경쟁 비교는 두지 않는다**(막힌 학습자 위축 방지).
- **옵트인 공개.** 사용자가 켜야 본인 활동이 피드에 올라간다. 기본값은 비공개이며, 공개 여부는 프로필 드롭다운의 공개 설정에서 토글한다.
- **추론 호출 없음.** 피드는 저장된 이벤트(통과·수료)만으로 렌더한다.
- 상단에 "이어서 하기" 한 줄을 둬 내 다음 행동이 묻히지 않게 한다. 강의실 발견·탐색은 둘러보기가 전담하므로 피드 안에 추천을 상시 끼우지 않는다(빈 상태의 권유는 예외).

### 7.2 둘러보기

공개 강의실을 검색·정렬(인기/최신)해 발견하는 화면. 강의실 카드(엠블럼/결과물 썸네일 + 제목 + 원작자 + 포크/통과/과제 수 + 포크 버튼)로 보여준다. 난이도 숫자·구간은 노출하지 않고 "입문용/심화" 거친 라벨까지만 허용한다(5.6절).

### 7.3 프로필

GitHub 프로필에 해당한다. 아바타·소개·깃허브 링크, **학습 열정 온도**(8장), 보유 스킬, 그리고 **내가 만든 공개 강의실** 그리드를 담는다. 피드·공개 뷰에서 작성자 이름을 누르면 도착하는 발견·포크의 허브다. 본인이 볼 때만 편집·비공개 강의실이 보이고, 타인이 볼 때는 공개 강의실만 보인다.

## 8. 동기와 진척 설계

자기주도 학습의 가장 큰 실패 원인은 방향을 몰라서가 아니라, 알면서도 중도에 그만두는 것이다. 이를 돕는 장치는 다음과 같다.

- **진척 가시화.** 스킬 트리를 화면에 그대로 보여준다. 완료·현재·남은 경로를 한눈에 드러낸다.
- **단계별 성취.** 각 마일스톤 완료가 분명한 성취로 느껴지게 한다. 그래픽스는 매 단계가 시각적 결과물을 내므로 이를 적극 활용한다.
- **복귀 유도.** 다음 과제는 직전 과제 통과 즉시 대기한다.
- **난이도 균형.** 스킬 트리 경로를 따라가되, 매 과제 결과(통과·반려·시도·강제통과)로 학습자가 수월한지 헤매는지를 가늠해 **다음 과제의 도움(스캐폴딩) 양을 조절**한다(5.6절). 정교한 난이도 숫자를 맞추는 것이 아니라 도움의 양으로 페이싱을 맞춘다. 내부 값(레벨 추정치·노드 티어·도움 양)은 사용자에게 노출하지 않는다.
- **균형 잡힌 피드백.** 리뷰는 엄격하되 통과한 성과는 분명히 인정한다.
- **활동 기록(잔디).** 트랙 대시보드에는 그 트랙 한정 활동 기록(잔디)을 두되, **연속(streak) 카운터는 두지 않는다.** 전체 트랙 합산 잔디는 내 강의실에 둔다.
- **학습 열정 온도.** 프로필에는 잔디 대신 **학습 열정 온도**를 둔다. 기준점 36.5°에서 시작해 학습 활동으로 천천히 올라가고, 활동이 없어도 급락하지 않고 완만하게만 내려가며 **하한은 기준점 근처**로 막는다. **연속·스트릭·끊김·리셋 개념과 표현을 쓰지 않으며**(리셋을 환기시키는 안심 문구조차 금지), **등수·타인 비교 없이** 내 상태만 보여준다. 낮을 때도 처벌·낙인 톤이 아니라 중립·격려 톤만 쓴다.

> 근거: 연속(streak) 시스템은 공백을 처벌해 강박과 죄책감을 유발하고, 한 번 끊기면 앱 자체를 멀리하게 만든다. "연속"이라는 축을 버리고 "빈도/누적 열정"이라는 축으로 바꾼다.

## 9. 온보딩

온보딩은 통짜 모달로 한 번에 보여주지 않고, **맥락별로 분산**한다.

- **가입 후속 셋업 마법사.** 환영 → GitHub 계정 연결 → 트레이너(말투) 선택 → 완료. 가입 시 1회만 거치고 끝나면 홈에 안착한다. 홈 위 팝업이 아니라 가입 흐름의 단계다.
- **AI 연결.** 1회성 절차가 아니라 상시 상태이므로 **계정 설정**에서 관리한다(연결 상태·모델·재확인·해제). 연결이 끊긴 채 추론이 필요한 시점(리뷰 요청·진단 제출·실라버스 생성)에는 그 자리에서 인라인 경고를 띄우고 해당 동작을 비활성화한다.
- **레포 등록.** 가입 절차가 아니라 **강의실 생성의 마무리 단계**에서 처리한다. GitHub 계정은 가입 때 연결됐으므로, 여기서는 "연결된 계정에 학습용 저장소를 새로 만들거나 기존 빈 저장소를 지정"하는 가벼운 단계다.

## 10. 실행 구조와 인프라

Toresen은 **호스팅 조율 계층**(조율·소셜·저장·표시)과 **연산 계층**(추론·빌드·렌더·유사도 비교)으로 나뉜다. 조율 계층은 모드와 무관하게 동일하고, 연산 계층의 실행 위치는 **연산 모드**로 정한다.

- **호스팅 서비스 (조율·상태·표시·소셜).** 강의실 정의·실라버스·과제 저장, 상태(단일 진실 소스), 웹 대시보드 호스팅, 커뮤니티 피드·프로필·둘러보기, GitHub 연동을 담당한다. 연산을 어디서 하든 이 계층은 바뀌지 않는다.
- **연산 모드 A — 클라이언트(BYO, 기본).** 커리큘럼·과제 생성·코드 리뷰 등 모든 LLM 작업과 빌드·렌더·캡처가 **사용자 본인의 LLM CLI·환경(예: Claude Code)에서 수행**된다. 서비스는 작업을 큐에 쌓고, 사용자 측 에이전트가 가져가 실행한 뒤 결과를 돌려준다. **비용은 전적으로 사용자 본인이 부담**한다. 에이전트가 저장소를 직접 읽어 변경 부분 밖의 맥락까지 리뷰에 활용한다. 다만 **골든과의 유사도 비교는 골든을 숨기기 위해 서버에서 수행**한다 — 클라이언트는 비밀이 아닌 자기 캡처만 올리고 판정만 돌려받으며, 이 가벼운 비교(GPU·추론 아님)는 BYO 부담에 해당하지 않는다(10장 종류 구분은 DESIGN 3.3). 이 모드에서 통과·점수 등 판정은 사용자 환경에서 산출되므로 **자가 보고**이며, 개인 진척 용도로만 신뢰한다.
- **연산 모드 B — 서버(중앙처리).** 커뮤니티 신뢰성을 중시하는 운영자가 선택한다. 빌드·렌더·유사도 비교, 그리고 LLM 추론을 **서비스 측 작업자가 운영자 부담 API/자원으로 수행**한다. 판정이 신뢰 경계 안에서 산출되고, **공개 통과 집계가 PR 머지가 아니라 이 게이트 판정을 세므로** 통과자 수 등 공개 신뢰 신호가 위조되지 않는다 — 학습자가 레포 소유권·머지 권한을 그대로 가진 채로도 성립한다. 대신 운영자는 임의 코드 샌드박싱·GPU·추론 비용과, 사용량 쿼터·어뷰즈 방어를 부담한다. **모드 B에서 서버에 두는 연산의 범위는 설정으로 정한다** — 최소 구성은 빌드·렌더·유사도 비교만 서버로 두어 공개 신호를 보증하고 LLM 리뷰는 클라이언트에 남기는 것이며, 최대 구성은 LLM 추론까지 서버에 두어 주관적 리뷰 판정 자체를 신뢰 신호로 삼는 것이다.
- **모드 전환.** A와 B는 **배포(운영자) 단위 설정**으로 전환하며, 둘은 같은 작업 큐 계약을 공유한다(DESIGN 2장). 작업의 실행 주체(사용자 기기 작업자 / 서비스 작업자)만 바뀌고 큐 인터페이스·상태·화면은 동일하다. 구현 기준값은 모드 A다.
- **GitHub은 표시·전달 계층.** 과제(Issue)와 리뷰(PR 코멘트)는 **전용 봇 계정**으로 게시되어 익숙한 코드 리뷰 UX를 제공한다. 서비스가 GitHub REST API를 직접 호출한다. 같은 내용을 웹 대시보드에 미러링한다.
- **보안.** 모드 A에서 LLM 에이전트는 셸 접근 권한을 가지므로 사용자 본인 기기에서만 실행하며, 자격 증명을 공용 서비스에 두지 않는다. 모드 B에서는 서비스가 사용자 코드를 격리된 샌드박스에서 빌드·실행하며, 사용자·강의실 단위 권한 격리·작업 큐 분리·자원 쿼터를 전제로 한다. 어느 모드든 공용 다중 사용자 서비스로서 사용자 간 권한 격리와 강의실 데이터 접근 제어를 둔다.

## 11. 향후 확장

- **포크 upstream 동기화.** 원본 강의실의 이후 변경을 포크가 선택적으로 받아오는 기능(현재 포크는 독립 스냅샷).
- **커뮤니티 상호작용.** 피드 항목에 대한 정해진 리액션(축하 등). 자유 메시지/포스팅은 모더레이션 부담이 커 더 뒤로 미룬다.
- **모드 B 운영 기능 강화.** 서버 연산의 사용량 쿼터·과금·어뷰즈 방어, 강의실/사용자별 모드 혼용(피드에서 인증 여부 배지로 구분).
- 그래픽스 외 도메인 및 다중 언어의 1차 지원 강화.
- 렌더링 검증을 환경 고정이 어려운 과제로 확대.
- 다중 사용자 운영을 위한 권한 격리, 작업 큐, 중복 방지 등 운영 안정성 강화.
