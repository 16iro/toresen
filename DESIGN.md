# Toresen 기술 설계

이 문서는 [SPEC.md](SPEC.md)("무엇을·왜")를 실제로 구현하기 위한 기술 구조("어떻게")를 정의한다. 큰 구조에서 세부로 내려가며 분할정복으로 기술한다.

---

## 1. 전체 구조

Toresen은 역할이 다른 세 부분으로 나뉜다.

- **컨트롤 영역(웹)** — 화면(대시보드), API, 상태 저장, 진행 상황 표시. "무엇을 돌릴지, 지금 상태가 어떤지"를 다룬다.
- **실행 영역(파이프라인)** — 제출물을 단계대로 처리하는 엔진과, 각 단계를 실제로 수행하는 작업자(agent).
- **GitHub** — 과제·제출·리뷰가 오가는 표시·전달 계층.

작업자는 두 종류이며, 필요한 자원이 있는 쪽에서 실행된다.

- **GPU 작업자(서버)** — 빌드, 렌더링, 화면 캡처, 렌더 유사도 비교.
- **LLM 작업자(사용자 기기)** — 커리큘럼 생성, 과제 생성, 코드 리뷰. 사용자 본인의 LLM CLI로 실행되어 LLM 비용을 사용자가 부담한다.

```
┌─ 컨트롤 영역 (웹) ─────────────┐
│  대시보드 · API · 상태 저장    │
└───────────────┬────────────────┘
                │
┌───────────────▼────────────────┐
│  실행 영역 (파이프라인 엔진)     │
│  단계를 작업자에게 분배          │
└──────┬───────────────────┬──────┘
     GPU                  LLM
┌──────▼────────┐   ┌──────▼─────────────┐
│ 서버 작업자    │   │ 사용자 기기 작업자  │
│ 빌드·렌더·캡처 │   │ 본인 CLI로 실행     │
│ ·유사도 비교   │   │ 커리큘럼·과제·리뷰  │
└────────────────┘   └─────────────────────┘

       GitHub  ·  과제(Issue) · 제출(PR) · 리뷰(코멘트)
```

## 2. 실행 아키텍처

### 2.1 컨트롤 영역과 실행 영역의 분리

화면을 띄우는 일과 파이프라인을 돌리는 일은 책임이 다르므로 분리한다. 컨트롤 영역은 상태와 표시를 맡고, 실행 영역은 실제 단계를 수행한다. (널리 쓰이는 CI/CD의 "컨트롤러 + 작업자" 구조와 같다.)

### 2.2 작업자와 작업 분배

실행 영역은 각 단계를 능력에 맞는 작업자에게 분배한다. GPU가 필요한 단계는 서버 작업자가, LLM이 필요한 단계는 사용자 기기의 작업자가 맡는다. 서버는 LLM 작업을 대기열에 쌓고, 사용자 기기의 작업자가 주기적으로 가져가 본인 CLI로 실행한 뒤 결과를 돌려준다. 이 방식이라 사용자 기기를 외부에 노출하지 않는다.

첫 버전에서는 두 작업자를 **같은 머신에 배치**해 실행하되, 둘 사이의 경계(작업 큐 계약)는 그대로 유지한다. 한 덩어리로 합치지 않으므로, 작업자를 나중에 다른 기기로 옮길 때는 연결 주소만 바꾸면 된다. 개발은 로컬호스트에서 하고, 검증되면 자체 호스팅 서버에 배포한다. LLM CLI는 실행되는 머신에서 사용자 본인 계정으로 동작하고, 사용자는 브라우저로 대시보드에 접속한다.

### 2.3 단계와 맞춤 파이프라인

각 단계는 컨테이너로 실행되는 독립 작업으로 정의한다. 어떤 단계를 어떤 순서로 거칠지는 **학습 주제(트랙)별 설정**으로 정한다. 그래픽스 트랙은 `빌드 → 렌더 → 유사도 비교 → 리뷰 → 채점`을 거치고, 렌더링이 없는 주제는 렌더 단계를 빼는 식이다. 새 주제를 추가할 때 설정과 단계 컨테이너만 더하면 되고 핵심 코드는 바뀌지 않는다.

이 파이프라인은 제출물을 채점하는 실행 흐름이며, 트랙을 준비할 때 정답 자료를 만드는 [코스 가이드라인 파이프라인](COURSE_GUIDELINE_PIPELINE.md)과는 다른 것이다.

### 2.4 구현 언어

컨트롤 영역과 파이프라인 엔진은 둘 다 조율·연결 역할이므로 **TypeScript로 통일**한다. 화면(프론트엔드)과 같은 언어라 한 가지만 익히면 된다. 무거운 계산이 필요한 단계(예: 렌더 유사도 비교)는 컨테이너 안에서 그 일에 가장 적합한 언어(예: Python)로 작성한다.

## 3. 데이터와 상태

### 3.1 무엇을 저장하나

| 항목 | 의미 |
|---|---|
| 트랙(Track) | 하나의 학습 주제 (예: 그래픽스 도장) |
| 스킬 노드(SkillNode) | 스킬 트리의 학습 단위. 선행 조건과 합격 기준을 가진다 |
| 과제(Assignment) | 노드에 대해 발급된 과제. 명세·기한·연결된 Issue |
| 제출(Submission) | 과제에 올린 PR. 브랜치·커밋·상태 |
| 리뷰(Review) | 제출에 대한 리뷰 결과 |
| 지적(Finding) | 리뷰 안의 개별 지적 항목 |
| 강제 통과 기록 / 진척·레벨 | override 사유, 노드별 진척, 현재 레벨 |

항목 간 연결은 GitHub의 Issue 번호와 PR 번호로 식별한다.

### 3.2 상태의 흐름

**제출은 다음 흐름을 따른다.**
```
발급 ──작업·브랜치──▶ 진행중 ──PR 생성──▶ 제출됨 ──리뷰 시작──▶ 리뷰중
리뷰중 ──통과──▶ 통과         (사용자가 머지하면 다음 과제 발급)
리뷰중 ──반려──▶ 같은 PR에 다시 push ──▶ 리뷰중   (이전 피드백 유지)
리뷰중 ──빌드/실행 실패──▶ 에러 ──재시도──▶ 리뷰중
어느 단계든 ── 강제 통과(override) ──▶ 통과(기록 남김)
```

**스킬 노드는 다음 흐름을 따른다.**
```
잠김 ──선행 조건 충족──▶ 열림 ──▶ 진행중 ──▶ 완료
모든 노드 완료 = 졸업
```

### 3.3 어디에 저장하나

"지금 무엇이 리뷰 중인지", "이 노드의 선행 조건이 끝났는지" 같은 조회가 잦으므로, 상태는 **SQLite 데이터베이스에 저장하고 이것을 단일 기준으로 삼는다.** GitHub에서 일어난 일(PR 머지·코멘트·강제 통과)은 폴링으로 감지해 데이터베이스에 반영하며, 최종 권위는 항상 데이터베이스가 가진다.

데이터베이스와 비밀 자료(접속 토큰, 정답 이미지)는 **서버의 로컬 디스크**에 있는 숨김 폴더 `.toresen/`에 두고 버전 관리에서 제외한다. 이 폴더는 학습자의 저장소 복제본에는 따라오지 않는다. 사람이 읽는 학습 계약서(`syllabus.md`)만 보이는 위치에 둔다.

GitHub에는 사람이 읽는 기록(과제·제출·리뷰·계약서)이 남으므로 상태의 일부는 이것으로 재구성할 수 있다. 그러나 점수·강제 통과 사유·정답 이미지·레벨처럼 GitHub에 없는 자료는 `.toresen/`에만 있으므로, 완전 복구를 위해 이 폴더를 따로 백업한다.

```
repo/
├── .toresen/          # 숨김 + 버전 관리 제외: 앱 내부 상태
│   ├── state.db       # 상태 데이터베이스 (단일 기준)
│   ├── golden/        # 정답 이미지 등 비밀 자료
│   └── config         # 접속 토큰 등
├── syllabus.md        # 사람이 읽는 학습 계약서 (보임)
└── ...                # 학습자 소스 코드
```

### 3.4 시작 시점

맨 처음에는 참고할 직전 리뷰가 없으므로, 진단용 과제 한두 개로 시작해 그 결과로 난이도를 맞춘다.

### 3.5 레벨 보정

학습자의 현재 레벨은 고정값이 아니라 성과에 따라 갱신되는 추정치다.

- **갱신 시점** — 매 과제가 끝날 때마다 그 결과(통과·반려, 점수, 시도 횟수, 강제 통과 여부)를 반영해 레벨을 다시 계산한다.
- **반영 경로** — 갱신된 레벨은 다음 과제 생성의 입력으로 들어가 난이도를 맞춘다. (시작 시점의 진단 과제로 초기값을 잡고, 이후 계속 보정한다.)
- 구체적인 계산식(무엇을 얼마나 반영할지)은 설정으로 분리해 조정할 수 있게 한다.

## 4. 리뷰

### 4.1 리뷰 요청과 실행

앱은 LLM 작업자에게 세 가지 일을 시킨다: 커리큘럼 생성, 과제 생성, 코드 리뷰. 리뷰는 **사용자 기기에서 저장소를 열어둔 채 CLI를 실행**하므로, 작업자가 코드를 직접 읽어 변경 부분 밖의 맥락까지 본다. 답은 아래의 정해진 형식으로 받도록 요청한다.

### 4.2 리뷰 결과 형식

코드 리뷰 작업의 출력은 아래 형식으로 고정한다(무엇을 리뷰하든 동일). 앱은 이 출력으로 코멘트를 달고 합격을 계산한다.

```jsonc
{
  "needs_rewrite": false,            // 근본부터 다시 짜야 하는가 (코멘트 방식 결정에 사용)
  "findings": [                      // 지적 목록 (없으면 빈 배열)
    {
      "file": "src/raycast.cpp",
      "line": 42,                    // 특정 줄이 없으면 null
      "dimension": "architecture",   // concept | convention | architecture | commit | performance … (루브릭이 더 추가 가능)
      "severity": "major",           // info | minor | major | critical
      "confidence": 0.8,             // 0.0 ~ 1.0
      "blocking": true,              // 이 지적 하나로 진행을 막을 수 있는가
      "title": "의존성 역전 위반",
      "body": "설명…",
      "evidence": "근거가 되는 실제 코드 인용"
    }
  ],
  "code_score": 72,                  // 0 ~ 100, 진척 표시용
  "action_items": ["다음으로 넘길 미해결 항목"],
  "summary": "대시보드용 요약 (PR 코멘트에는 달지 않음)"
}
```

**최종 통과/반려는 이 출력에 들어 있지 않다.** 앱이 4.4의 게이트로 계산한다 — 이 출력의 지적(`blocking`·`confidence`) + 렌더 검증 결과(4.5) + 기계적 1차 검사를 합쳐 정한다. 그래서 LLM은 판정을 내리지 않고 근거 있는 지적만 제공하며, 판정 권한은 앱 한 곳에 모인다(이중 판정 방지).

렌더 검증의 유사도 결과(SSIM)는 별도 단계(4.5)의 산출물이며 합격 계산 시 함께 반영된다.

### 4.3 코멘트 게시 방식

코멘트에는 칭찬이나 요약 없이 **지적만** 단다(통과 인정 등은 대시보드에서 보여준다). 지적의 무게에 따라 세 방식으로 나누고, **무거운 쪽부터** 판단한다.

1. **근본을 다시 짜야 하는 경우** — 큰 틀만 지적하고 재작성을 권한다. 줄 단위 코멘트는 생략한다.
2. **중간 정도인 경우** — 문제 하나당 코멘트 하나를, 해당 줄에 단다.
3. **가볍고 적은 경우** — 코멘트 하나에 모아서 단다.

"재작성이 필요한가"는 리뷰가 판단하고, 1이 아니면 지적의 개수에 따라 2와 3을 고른다.

### 4.4 합격과 머지

평가는 두 단계를 거친다.

**1단계 — 기계적 1차 검사.** 딱 떨어지는 사실은 LLM이 아니라 도구가 먼저 판정한다. 빌드 성공 여부, 금지된 기능 사용 여부(과제별 금지 목록을 단순 패턴으로 검사 — 예: 특정 단계 전까지 3D API 사용 금지), 그리고 제공된 테스트가 있으면 그 통과 여부다. 여기서 막히면 LLM 리뷰를 돌리지 않고 즉시 반려한다. 추론 비용을 아끼고 피드백도 명확하다. (첫 버전은 빌드·금지 기능 검사가 필수이고, 테스트는 선택이다.)

**2단계 — 게이트 판정.** 1차 검사를 통과하면 코드 리뷰와 렌더 검증으로 합격을 정한다. 확신이 높고 진행을 막는 지적이 있거나 렌더 유사도가 기준에 못 미치면 반려, 그렇지 않으면 통과다. 확신이 낮은 지적은 코멘트로 남기되 진행을 막지 않는다. 점수는 표시용이며 합격을 직접 좌우하지 않는다.

GitHub에서 봇은 통과면 승인을, 반려면 변경 요청을 의견으로 표시한다. **실제 통과는 사용자가 직접 PR을 머지할 때 일어난다.** 머지되면 그 과제 코드가 main에 쌓이고(엔진이 단계별로 자란다) 다음 과제가 발급된다. 봇이 변경 요청을 했더라도 사용자가 강제 통과(override)하면 승인 없이 머지할 수 있으며, 이 경우 기록을 남긴다.

### 4.5 렌더 검증 (시각 과제)

화면 결과물이 있는 과제는 코드와 별도로 렌더링 결과를 검증한다.

- **환경 고정** — 과제는 렌더링에 필요한 환경(모델·텍스처·카메라·조명·해상도)을 정해준다. 학습자는 알고리즘만 구현하므로, 올바른 구현의 결과 화면은 하나로 정해진다.
- **자동 캡처** — 학습자 프로그램은 정해진 화면을 한 장(애니메이션은 정해진 시점들의 여러 장) 파일로 저장하고 종료하는 모드를 가진다. 이 뼈대는 스타터 템플릿에 포함되며, 캡처는 서버에서 수행한다.
- **유사도 비교** — 캡처한 화면을 정답 이미지와 구조적 유사도(SSIM)로 비교한다. GPU·연산 차이로 완전 일치는 불가능하므로, 1.0이 아니라 과제별로 정한 기준값(예: 0.95 이상)으로 판정한다.
- **정답 이미지의 출처** — 비교 기준이 되는 정답 이미지는 [코스 가이드라인 파이프라인](COURSE_GUIDELINE_PIPELINE.md)이 생성·검증해 숨겨 보관한다. 학습자에게는 보이지 않는다.
- **환경 정의의 단일 출처** — 카메라·모델·해상도·캡처 방식은 과제마다 하나의 정의로 두고, 학습자 스타터 템플릿·정답 생성·서버 캡처가 모두 같은 정의를 쓴다. 셋이 어긋나면 비교가 무의미해지므로 한 곳에서 정의해 공유한다.

렌더 검증 결과는 코드 분석과 함께 합격 판정(4.4)에 반영된다.

### 4.6 실패 처리

답이 형식에 맞지 않으면 정해진 횟수만큼 다시 요청한다. CLI 오류나 시간 초과는 잠시 쉬었다 재시도하고, 끝내 실패하면 조용히 넘기지 않고 "리뷰 실패"로 표시한다. 같은 제출(같은 커밋)에 대한 같은 작업은 한 번만 처리한다. 빌드나 렌더 캡처가 실패하면(예: 화면을 못 잡거나 빈 화면) 같은 방식으로 "검증 실패"로 표시하고 사용자에게 알린다.

### 4.7 무엇을 평가할지는 편집 가능

리뷰 *형식*은 고정이지만, **무엇을 평가할지는 사용자가 바꾼다.** 개념만 가볍게 보는 방식부터, 아키텍처·코드 컨벤션·커밋 메시지까지 모두 보는 숙련 방식까지, 학습 목적에 맞춰 트랙별로 설정한다. 이 설정은 리뷰 요청에 끼워 넣는 것이라 핵심 코드를 건드리지 않는다.

## 5. GitHub 연동

### 5.1 두 정체성: 학습자와 봇

학습자는 본인의 GitHub 계정으로 코드를 올리고 최종 머지를 한다. 리뷰어는 **별도의 봇 계정**이다. 봇이 별도 계정이어야 하는 이유는, GitHub에서는 자기 PR을 자기가 승인할 수 없기 때문이다. 봇이 승인·변경 요청을 표시하려면 학습자와 다른 계정이어야 한다.

봇의 GitHub 프로필(이름·아바타)은 **하나로 통일**한다. 리뷰어의 표시 페르소나(이름, 아바타, 말투)는 리뷰를 미러링하는 **웹 대시보드에서 사용자가 직접 꾸민다.** GitHub 계정을 사용자마다 따로 만들지 않으므로 계정 관리나 정책 문제가 없고, 풍부한 페르소나 경험은 웹 쪽이 담당한다.

### 5.2 봇 인증

봇 계정의 접속 토큰을 발급해 숨김 폴더 `.toresen/config`에 보관한다. 서버는 이 토큰으로 과제(Issue) 생성, 리뷰 코멘트, 승인·변경 요청을 수행한다. 토큰 권한은 해당 저장소 범위로 제한한다.

토큰이 만료되거나 무효가 되면 외부 호출이 조용히 실패할 수 있으므로, 인증 실패를 감지하면 해당 작업을 멈추고 대시보드에 "토큰 만료"로 표시해 재등록을 유도한다.

### 5.3 학습용 저장소

학습은 **트랙별 스타터 템플릿**에서 시작한다. 템플릿은 폴더 구조와 빌드 설정 같은 뼈대만 담으며 정답 코드는 포함하지 않는다. 이 템플릿에서 학습자 계정에 저장소를 만들고, 봇 계정을 협업자로 추가하면 봇이 리뷰를 달 수 있다. 저장소는 비공개여도 된다. 만들어진 저장소는 앱에 등록되어 이후 모든 흐름과 연결된다. 처음 한 번의 저장소 준비는 사용자가 직접 하고, 자동화는 이후 과제로 둔다. 스타터 템플릿은 트랙을 준비할 때 만들어지며, 화면 캡처 모드와 과제별 환경 정의(4.5)를 정답 생성 쪽과 공유한다.

### 5.4 트리거

앱은 외부에서 들어오는 신호(웹훅)를 받지 않는다. 새 제출은 서버가 GitHub를 주기적으로 확인(폴링)하거나 대시보드의 제출 동작으로 감지한다. 봇은 모든 작업을 나가는(outbound) 호출로만 수행한다.

## 6. 시작과 온보딩

### 6.1 처음 한 번의 설정

앱을 처음 실행하면 다음 1회성 설정을 거친다.

- **LLM CLI 연결** — 사용자의 CLI 위치를 지정하고 로그인 상태를 확인한다.
- **봇 토큰 등록** — 봇 계정의 접속 토큰을 입력받아 `.toresen/config`에 저장한다.
- **본인 GitHub 연결** — 사용자 계정을 연결한다.

설정 도중 CLI를 찾지 못하거나 토큰이 유효하지 않으면 그 자리에서 알리고 다음으로 진행하지 않는다.

### 6.2 트랙 시작

설정이 끝나면 학습 주제(트랙)를 시작한다. 트랙을 시작하면 스타터 템플릿에서 학습용 저장소를 만들고(5.3), 커리큘럼을 협상해 학습 계약서를 확정한 뒤, 진단 과제로 첫 단계를 연다(3.4).

첫 버전에서는 봇 계정과 학습용 저장소를 사용자가 한 번 직접 만들어 앱에 등록하고, 그 외 설정 저장과 연결은 앱이 자동으로 처리한다. 생성 자동화는 이후 과제로 둔다.
