# Toresen 기획서

## 1. 개요

Toresen은 LLM을 **엄격한 시니어 리뷰어**로 활용하는 자기주도 하드코어 트레이닝 시스템이다.

가장 중요한 원칙은 **AI가 코드를 대신 작성하지 않는다**는 것이다. AI는 학습자의 코드를 리뷰하고, 다음 과제를 출제하고, 개념을 설명한다. 그러나 구현 코드나 복사해 쓸 수 있는 템플릿은 제공하지 않는다. 학습자는 스스로 코드를 작성하며 자신의 한계를 넘어선다.

기존 코딩 학습 도구가 "막히면 답을 보여주는" 방향이라면, Toresen은 "막혀도 스스로 풀게 하되, 시니어가 옆에서 매섭게 피드백하는" 도장(dojo)에 가깝다.

## 2. 범위

첫 번째 버전은 **그래픽스 프로그래밍 훈련**에 집중한다. 학습자는 3D 그래픽스의 역사적 이정표를 직접 클론하며 단계적으로 올라간다 — 소프트웨어 레이캐스팅(울펜슈타인) → 공간 분할과 2.5D(둠) → 순수 3D와 셰이더 → Vulkan 기반 모던 렌더링 엔진.

| 구분 | 내용 |
|---|---|
| 도메인 | 그래픽스 (단일) |
| 언어 | C++ 또는 Rust (단일) |
| 대상 학습자 | Git과 터미널에 익숙한 중급 이상 개발자 |
| 인프라 비용 | 클라우드 추론 비용 없음 (8장 참고) |

## 3. 핵심 원칙

1. **AI 코드 생성 금지.** 리뷰·과제 출제·개념 설명만 한다. 아키텍처 원칙이나 트레이드오프 설명은 허용하지만, 완성된 함수 본문이나 복사 가능한 구현 블록은 제공하지 않는다.
2. **로컬 우선.** 무거운 처리(빌드, 리뷰 추론)는 학습자의 로컬/홈 환경에서 수행한다. 휴대용 기기에서는 코드 작성과 `git push`만으로 사이클이 돌아간다.
3. **에이전트가 저장소를 직접 읽는다.** 리뷰 시 변경분만 보는 것이 아니라, 리뷰 엔진이 로컬 저장소 전체를 읽어 호출 관계·의존성·빌드 결과까지 직접 확인한다.

## 4. 동작 흐름

```
레벨 진단
   │
   ▼
커리큘럼 합의  ──►  syllabus.md (학습 계약서)
   │
   ▼
┌─────────────── 훈련 루프 (반복) ───────────────┐
│  과제 발급(Issue) → 코딩 → Pull Request 제출   │
│        → 리뷰 → 통과 시 다음 과제              │
│        → 반려 시 이전 피드백 유지하며 재시도   │
│        → 교착 시 강제 통과(override) 비상구    │
└────────────────────────────────────────────────┘
```

역할 분담은 다음과 같다.

- **학습자(사람):** 학습 방향 설정, 코드 작성, 렌더링 화면 직접 확인, 리뷰에 대한 비판적 수용과 반론.
- **AI:** 다음 과제 생성, 코드 리뷰, 통과/반려 판정.

## 5. 커리큘럼 설계

학습 성패는 "무엇을 만들지"를 처음에 옳게 정하는 데 달려 있다. 학습자가 해당 분야를 아직 모르는 상태에서 범위를 정하면 잘못된 커리큘럼이 나오기 쉽다. Toresen은 이를 두 단계로 방지한다.

### 5.1 레퍼런스 기반 커리큘럼 생성

LLM의 내부 지식에만 의존하면 일반론적이거나 학습자 의도와 다른 커리큘럼이 나온다. 따라서 **기준 문서(reference)를 근거로** 커리큘럼을 구성한다. 세 가지 소스를 지원하며, 각각 신뢰도가 다르다.

| 소스 | 처리 방식 | 신뢰도 |
|---|---|---|
| 코드베이스 (예: 둠 오픈소스) | 모듈 구조와 의존 관계를 추출하되, 인터페이스와 역할만 사용한다. 실제 구현 코드는 학습자에게 노출하지 않는다(정답지 역할). | 높음 |
| 논문 (예: SIGGRAPH 자료) | 핵심 알고리즘과 함께, 논문이 전제하는 선수 지식까지 학습 항목으로 끌어낸다. | 중간 |
| 강의 슬라이드 (예: GDC/CEDEC) | 슬라이드 이미지를 멀티모달로 읽고, 가능하면 발표 트랜스크립트를 함께 사용한다. 정보 공백이 많다. | 낮음 |

생성된 모든 학습 항목에는 **출처와 신뢰도를 함께 표시**한다. 학습자는 최종적으로 이 항목들을 검토하고 가지치기한다.

### 5.2 전체 지형 제시 후 범위 합의

학습자에게 범위를 묻기 전에, 시스템이 **해당 주제의 전체 기술 지형을 먼저 제시**한다. 예를 들어 울펜슈타인이라면 "핵심 기술은 레이캐스팅·텍스처 매핑·스프라이트·맵 포맷·입력·오디오이며, 각각은 이런 개념이고 난이도는 이 정도"라고 펼쳐 보인다.

학습자는 이 지형을 본 뒤에 범위를 정한다("레이캐스팅과 텍스처까지만, 오디오는 직접 구현 대신 단순화"). 모르는 상태에서 찍는 것이 아니라, 알고 선택하게 한다.

### 5.3 스킬 트리와 학습 계약서

합의 결과는 단순 목록이 아니라 **의존성 그래프(스킬 트리)**로 표현한다. 선행 개념이 후행 과제의 전제가 되도록 순서를 잡는다(예: 벡터 수학 → 공간 분할 순회).

이 그래프는 `syllabus.md`로 저장소에 고정되며, 이후 모든 리뷰가 참조하는 **학습 계약서**가 된다.

### 5.4 레벨 진단

초기 진단은 간단한 자가진단 설문으로 학습자를 **초급/중급/고급의 거친 범주**로만 분류한다. 정밀한 단일 점수는 신뢰할 수 없으므로 사용하지 않는다.

실질적인 난이도 보정은 **초기 진단 과제 한두 개의 실제 수행 결과**로 이루어지며, 이후에도 학습자의 성과에 따라 동적으로 조정된다.

## 6. 리뷰 시스템

### 6.1 리뷰 입력

리뷰 엔진은 변경된 코드 조각만 보지 않는다. 로컬 저장소 전체에 접근하여 **변경된 부분 밖의 맥락**(호출하는 쪽, 의존하는 파일, 빌드 결과)까지 직접 확인한다. 아키텍처 리뷰는 본질적으로 변경 파일 바깥의 관계를 봐야 하기 때문이다.

### 6.2 코드 분석 (주 평가축)

평가의 중심은 코드 분석이다. 아키텍처와 설계 — 모듈 분리, 의존성 방향, 메모리 레이아웃, 언어 관용구, 금지된 API 사용 여부 — 를 검증한다.

이 중 명확히 판정 가능한 항목(컴파일 성공 여부, 금지 API 사용, 테스트 통과)은 LLM이 추측하지 않고 **컴파일러·정적 분석 도구·테스트가 1차로 결정**한다.

### 6.3 렌더링 검증 (보조 평가축)

그래픽스는 코드가 옳아 보여도 실제 화면이 잘못될 수 있다. 이를 잡기 위해 렌더링 결과를 검증한다.

- **비중:** 렌더링 결과물이 있는 과제는 코드 분석 70% / 렌더링 검증 30%로 평가한다. 화면 결과가 없는 순수 아키텍처 과제는 코드 분석 100%로 평가한다.
- **입력 고정(soft-lock):** 과제는 렌더링에 필요한 환경 — 모델, 텍스처, 카메라 위치, 조명 — 을 엄격히 지정한다(이 지정은 가이드라인일 뿐 구현 코드를 주는 것은 아니다). 학습자는 셰이더와 알고리즘만 구현한다. 환경이 고정되면 올바른 구현의 기대 출력은 하나로 정해진다.
- **자동 판정:** 고정된 환경에서 정해진 카메라로 화면을 캡처하고(애니메이션은 연속 프레임), 미리 준비한 정답 이미지와 **구조적 유사도(SSIM 등) 기준으로 비교**한다. GPU·부동소수점 차이로 픽셀 단위 완전 일치는 불가능하므로, 임계값 기반의 지각적 유사도로 판정한다.
- **멀티모달 보조:** 환경 고정이 어려운 과제나 "무엇이 잘못되었는지" 설명이 필요한 경우에만 멀티모달 판정을 보조로 사용한다.

### 6.4 리뷰 신뢰성

단정적인 리뷰어가 확인되지 않은 내용을 자신 있게 지적하면, 학습자가 옳은 코드를 의심하거나 비상구를 남용하게 된다. 이를 막기 위해:

- **근거 강제:** 모든 지적은 실제 코드 위치(파일·라인)를 인용해야 한다. 근거를 가리킬 수 없으면 지적할 수 없다.
- **태도 분리:** 취향과 설계 원칙에는 단호하게 의견을 내되, 확인하지 못한 사실에는 신중하게 표현한다.
- **신뢰도 게이팅:** 근거가 있고 확신이 높은 지적만 과제 진행을 막을 수 있다. 확신이 낮은 지적은 진행을 막지 않는 선택적 제안으로 남긴다.

### 6.5 리뷰 일관성

훈련 루프가 길어져도 리뷰가 흔들리지 않도록, 매 리뷰마다 **학습 계약서(`syllabus.md`), 직전 리뷰 요약, 미해결 항목**을 함께 참조한다. 이로써 이미 합의된 범위를 나중에 다시 지적하는 모순을 방지한다. 히스토리가 길어지면 마일스톤 단위로 요약해 압축한다.

### 6.6 강제 통과 처리

학습자가 교착 상태에서 강제 통과(override)를 사용하면, 해당 제출물은 미완성 상태로 통과된다. 이런 제출물이 이후 리뷰의 참고 맥락에 섞이면 잘못된 코드가 모범으로 학습될 수 있으므로, **강제 통과된 제출물은 이후 리뷰의 참고 대상에서 제외**하고 그 사유를 기록한다.

## 7. 동기와 진척 설계

학습 범위와 종료 지점은 스킬 트리(5.3절)가 이미 정의한다. 이 장은 학습자가 그 경로를 **중간에 포기하지 않고 끝까지 완주**하도록 돕는 장치를 다룬다. 자기주도 학습의 가장 큰 실패 원인은 방향을 몰라서가 아니라, 알면서도 중도에 그만두는 것이기 때문이다.

- **진척 가시화.** 스킬 트리를 그대로 화면에 보여준다. 완료한 항목, 현재 항목, 남은 경로를 한눈에 드러내어 추상적인 목표를 매일 확인 가능한 진척으로 바꾼다.
- **단계별 성취.** 최종 목표(렌더링 엔진)는 멀기 때문에, 각 마일스톤 완료가 분명한 성취로 느껴지도록 한다. 그래픽스 훈련은 매 단계가 눈에 보이는 결과물(렌더링된 화면)을 내므로, 진척이 시각적으로 드러나는 점을 적극 활용한다.
- **복귀 유도.** 다음 과제는 직전 과제 통과 즉시 Issue로 발급되어 대기한다. 다음 할 일이 항상 명확히 놓여 있어 학습자가 다시 돌아오게 만든다.
- **난이도 균형.** 과제가 너무 어려우면 이탈하고 너무 쉬우면 지루해진다. 적응형 레벨 보정(5.4절)으로 학습자의 실제 성과에 맞춘 난이도를 유지한다.
- **균형 잡힌 피드백.** 리뷰는 엄격하되, 통과한 성과는 분명히 인정한다. 비판만 누적되지 않도록 한다.

## 8. 실행 구조와 인프라

Toresen은 상용 클라우드에 의존하지 않고, 사용자가 직접 운영하는 환경에서 실행된다. 작업은 필요한 자원의 종류에 따라 두 곳으로 나뉜다. 첫 버전에서는 이 두 역할이 한 대의 자체 호스팅 머신에서 함께 실행되며, 둘을 다른 기기로 나누는 것은 향후 확장이다.

- **자체 호스팅 서버 (GPU·상태·표시 담당).** 사용자가 보유한 서버가 빌드, 렌더링과 화면 캡처, 렌더링 유사도 비교(SSIM), 상태 저장(단일 진실 소스), 웹 대시보드 호스팅, GitHub 연동을 담당한다. 빌드·렌더 환경은 컨테이너로 표준화한다.
- **사용자 기기 (LLM 추론 담당).** 커리큘럼 생성·과제 생성·코드 리뷰 등 **모든 LLM 작업은 사용자 본인의 LLM CLI(예: Claude Code)로 사용자 기기에서 수행**된다. 서버는 LLM 작업을 큐에 쌓고, 사용자 기기의 에이전트가 이를 가져가 실행한 뒤 결과를 돌려준다. 따라서 **LLM 비용은 전적으로 사용자 본인이 부담**하며 서버는 추론 비용을 지지 않는다. 에이전트가 로컬 저장소를 직접 읽어, 변경 부분 밖의 맥락까지 리뷰에 활용한다.
- **GitHub은 표시·전달 계층.** 과제(Issue)와 리뷰(Pull Request 코멘트)는 **전용 봇 계정**으로 GitHub에 게시되어 익숙한 코드 리뷰 UX를 제공한다. 서버가 GitHub REST API를 직접 호출(outbound)하며 외부에서 들어오는 웹훅에 의존하지 않는다. 같은 리뷰 내용을 웹 대시보드에도 미러링한다.
- **보안.** LLM 에이전트는 셸 접근 권한을 가지므로 사용자 본인 기기에서만 실행하며, 자격 증명을 공용 서버에 두지 않는다. 단일 신뢰 사용자 환경을 전제로 한다.

## 9. 향후 확장

- **분리 실행 구성.** 휴대용 기기에서 코드를 작성하고 빌드·리뷰는 홈 데스크톱에서 수행하는 구성. 홈 데스크톱이 새 제출을 주기적으로 확인(폴링)해 처리하므로, 데스크톱이 꺼져 있어도 켜질 때 밀린 작업을 따라잡는다.
- 그래픽스 외 도메인 및 다중 언어 지원.
- 본격 게이미피케이션: 연속 학습(streak), 성취 배지 등.
- 렌더링 검증을 환경 고정이 어려운 과제로 확대.
- 다중 사용자 운영을 위한 권한 격리, 작업 큐, 중복 방지 등 운영 안정성 강화.
